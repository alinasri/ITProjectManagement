🎯 Complete Walkthrough

Two scenarios, end-to-end, no gaps. Follow every line of code from user action to UI update. This lesson ties together everything from the previous 10 lessons.

Scenario A — Login and Navigate to Purchases

A purchase_admin user opens the app for the first time today and logs in.

Step A1: Browser Loads the App

The browser requests http://localhost:5173. Vite serves index.html:

<div id="root"></div>
<script src="/src/main.jsx"></script>

The browser downloads and runs main.jsx.

Step A2: React Mounts

// main.jsx
createRoot(document.getElementById('root')).render(
  <StrictMode><App /></StrictMode>
);

React starts rendering the component tree from App. The render order: App → AuthProvider → BrowserRouter → Routes.

Step A3: AuthProvider Initializes

// AuthContext.jsx — initial state
user    = null    ← no user yet
loading = true    ← checking...

The useEffect is scheduled to run after this first render. React continues rendering.

Step A4: Route Matching

Current URL is /. React Router matches the path="/" route, which renders RoleRedirect:

function RoleRedirect() {
  const { user, loading } = useAuth();
  if (loading) return <FullPageSpinner />;  ← renders spinner
  ...
}

Since loading=true, the spinner is rendered. The user sees a spinning circle.

Step A5: Auth Check Completes

React runs the useEffect in AuthProvider. It calls auth.me() → GET /api/auth/me. There is no session cookie (first visit). The backend responds 401.

// AuthContext.jsx
.catch(() => setUser(null))    ← user stays null
.finally(() => setLoading(false))  ← loading becomes false

Calling setLoading(false) triggers a re-render of every component that uses useAuth().

Step A6: Re-render → Redirect to Login

RoleRedirect re-renders with the new state:

user    = null
loading = false
// Hits this branch:
if (!user) return <Navigate to="/login" replace />;

React Router updates the URL to /login and renders LoginPage (wrapped in PublicOnly). PublicOnly also checks auth — since user=null and loading=false, it just renders children (the login form).

The user now sees the login form.

Step A7: User Types and Submits

The user types their username and password. Each keystroke calls:

onChange={e => setUsername(e.target.value)}
onChange={e => setPassword(e.target.value)}

State updates → re-renders → input shows the typed text.

User clicks "ورود". The form's onSubmit fires → handleSubmit(e) runs:

e.preventDefault();   ← prevent browser form submission (page reload)
setLoading(true);      ← button shows "در حال ورود..."
const user = await login(username, password);

Step A8: AuthContext.login Runs

// AuthContext.jsx
const login = async (username, password) => {
  const r = await auth.login({ username, password });
  // POST /api/auth/login → 200 OK
  // response: { user: { id: 7, role: 'purchase_admin', ... } }
  // backend also sets httpOnly cookie: token=eyJ...
  setUser(r.data.user);   ← global user state updated!
  return r.data.user;
};

Step A9: Navigate to Purchases

Back in handleSubmit:

const user = await login(...);
// user.role === 'purchase_admin'
// doesn't match super_admin or it_head, so falls through to:
navigate(`/section/${user.section_id}`);
// Wait — purchase_admin doesn't have section_id. They'd hit the else branch:
// Actually the LoginPage's simple navigation logic goes to /section/:sectionId
// for non-admin users. For purchase_admin, AuthContext login returns the user,
// and the navigate in handleSubmit sends them to their home page.

After setUser() is called in AuthContext, all components using useAuth() re-render. But navigate() in LoginPage runs immediately and changes the URL, so the new page component renders, not the login page.

Step A10: ProtectedLayout and Layout Mount

URL is now /purchases. React Router renders the nested route tree. The outer route matches → renders ProtectedLayout:

if (loading) return <FullPageSpinner />;     ← loading=false, skip
if (!user)   return <Navigate to="/login" />; ← user is set, skip
if (user.must_change_password) return ...;   ← false, skip
return (
  <SectionsProvider>   ← sections fetch starts here
    <Layout />           ← sidebar + Outlet
  </SectionsProvider>
);

SectionsProvider mounts → fetches sections → setSections(data) → Sidebar re-renders with section links.

Layout renders: Sidebar on the left, Outlet on the right.

Step A11: RequireAuth and PurchasesPage Mount

The inner route's RequireAuth renders:

roles = ['super_admin', 'it_head', 'purchase_admin']
user.role = 'purchase_admin'
roles.includes('purchase_admin') → true
return <Outlet />;   ← render PurchasesPage

PurchasesPage mounts. Its useState initializes:

list       = []
loading    = true
showArchived = false
searchTerm = ''
...

Step A12: PurchasesPage Fetches Data

// PurchasesPage.jsx
const fetchAll = useCallback(async () => {
  const r = await purchasesApi.list({});  ← GET /api/purchases
  setList(r.data);                         ← update state with records
}, [showArchived]);                          ← showArchived=false, so no filter

useEffect(() => { fetchAll(); }, [fetchAll]);  ← fires after first render

The backend responds with an array of purchase objects. setList(r.data) triggers a re-render. The table renders with all the purchases.

The user now sees the Purchases page with the sidebar, the stat cards, and the table of purchases.

Scenario B — User Edits a Purchase Record

The purchase_admin clicks the edit pencil icon on a purchase row.

Step B1: Click Triggers startEdit

// PurchasesPage.jsx
const startEdit = (row) => {
  setEditingId(row.id);   ← mark this row as being edited
  setEditRow({
    title: row.title,
    status: row.status,
    supplier: row.supplier,
    amount: row.amount,
    purchase_date: row.purchase_date,
    description: row.description,
    section_ids: row.sections?.map(s => s.id) ?? [],
  });
};

Two state updates fire. React batches them and does one re-render. The row with id === editingId switches to edit mode (showing inputs instead of text).

Step B2: User Modifies Fields

The user changes the status dropdown. Each change calls:

onChange={e => setEditRow(prev => ({ ...prev, status: e.target.value }))}

The functional update form prev => ({ ...prev, ... }) ensures we keep all existing fields and only change status. React re-renders — the dropdown shows the new value.

Step B3: User Clicks Save

const saveEdit = async (id) => {
  await purchasesApi.update(id, { ...editRow });  ← PUT /api/purchases/42
  setEditingId(null);   ← exit edit mode
  fetchAll();           ← re-fetch the whole list to show updated data
};

The backend updates the record in the database and returns the updated purchase. fetchAll() re-fetches the entire list. setList(r.data) fires, React re-renders the table with fresh data.

The row now shows the updated status. The UI is in sync with the database.

The Full Mental Model

After reading these two scenarios, you should be able to answer these questions about any feature in this app:

QuestionWhere to look
How does the app know who is logged in?context/AuthContext.jsx — user state
How does a page fetch its data?useEffect + useCallback in the page component
How does data reach the backend?api/index.js → api/client.js → axios → Vite proxy → Express
How does the UI update after a change?Setter function called → React re-renders → new JSX reflects new state
How does navigation work?useNavigate() or <Navigate /> → React Router → URL changes → route match → new component
How does the sidebar know which links to show?user.role from AuthContext drives conditional rendering in Sidebar
How do pages share the sections list?SectionsContext — fetched once, shared via useSections()
What happens when the session expires?Any API call → 401 → interceptor → window.location.href = '/login'

The core loop of every React feature:
1. User does something
2. Event handler runs
3. State changes (setState)
4. React re-renders
5. New JSX reflects new state
6. User sees the updated UI

Everything in this app — from login to deleting a record to filtering a table — is some variation of this loop.
