🌐 Context API

Context is React's solution to sharing data across the entire component tree without passing it through every intermediate component. This project uses it for the logged-in user and the sections list.

The Problem: Prop Drilling

Imagine you need to show the current user's name in the Sidebar. The Sidebar is buried deep in the component tree:

App └── ProtectedLayout └── Layout ← needs to pass user down ├── Sidebar ← needs user.username │ └── SidebarLink └── main └── PurchasesPage ← also needs user.role └── some component that also needs user...

Without context, you'd have to pass user as a prop to ProtectedLayout, which passes it to Layout, which passes it to Sidebar, etc. This is called prop drilling. It's painful — every intermediate component has to accept and forward the prop even if it doesn't use it.

Context solves this by creating a "broadcast channel". The AuthProvider puts the user data onto this channel, and any component anywhere in the tree can tune in with useAuth().

The Three Parts of Context

1. createContext — Create the channel

const AuthContext = createContext(null);
// null is the default value — used when a component is outside the Provider

2. Provider — Broadcast the data

<AuthContext.Provider value={{ user, setUser, login, logout, loading }}>
  {children}   ← every component nested inside can access these values
</AuthContext.Provider>

3. useContext — Tune in to the channel

const { user, loading } = useContext(AuthContext);
// Shorthand via the custom hook:
const { user, loading } = useAuth();

AuthContext — Deep Dive

Here is the full AuthContext.jsx with detailed comments:

// context/AuthContext.jsx
import { createContext, useContext, useState, useEffect, useRef } from 'react';
import { auth } from '../api';

// Step 1: Create the context object — the "channel"
const AuthContext = createContext(null);

// Step 2: The Provider component — holds and shares auth state
export function AuthProvider({ children }) {
  const [user, setUser]       = useState(null);    ← null = not logged in
  const [loading, setLoading] = useState(true);    ← true = checking session
  const checked = useRef(false);

  // On first mount: ask the backend "who am I?"
  useEffect(() => {
    if (checked.current) return;   ← StrictMode guard
    checked.current = true;

    auth.me()
      .then(r  => setUser(r.data))   ← logged in: set user
      .catch(() => setUser(null))    ← not logged in: user stays null
      .finally(() => setLoading(false));  ← done either way
  }, []);

  // login() — called by LoginPage on form submit
  const login = async (username, password) => {
    const r = await auth.login({ username, password });
    setUser(r.data.user);     ← update global user state
    return r.data.user;       ← return user so LoginPage can check role
  };

  // logout() — called by Sidebar's logout button
  const logout = async () => {
    await auth.logout();    ← tell backend to clear the cookie
    setUser(null);          ← clear local user state → triggers redirect to /login
  };

  // Step 3: Provide the values to all children
  return (
    <AuthContext.Provider value={{ user, setUser, login, logout, loading }}>
      {children}
    </AuthContext.Provider>
  );
}

// Custom hook — shorthand for useContext(AuthContext)
export const useAuth = () => useContext(AuthContext);

What does "loading" protect against?

When the app first loads, user is null and loading is true. Without the loading flag, every protected route would immediately redirect to /login — even if the user IS logged in, just because the auth check hasn't finished yet.

The ProtectedLayout in App.jsx checks loading first:

if (loading) return <FullPageSpinner />;   ← wait for check
if (!user)   return <Navigate to="/login" />; ← now safe to check

SectionsContext — A Simpler Example

SectionsContext is simpler than AuthContext — it just loads and shares the list of sections:

// context/SectionsContext.jsx
const SectionsContext = createContext([]);  ← default: empty array

export function SectionsProvider({ children }) {
  const [sections, setSections] = useState([]);

  // refresh is memoized so it can be a safe useEffect dependency
  const refresh = useCallback(() => {
    sectionsApi.list().then(r => setSections(r.data)).catch(() => {});
  }, []);  ← no deps: refresh never changes identity

  useEffect(() => { refresh(); }, [refresh]);  ← fetch once on mount

  return (
    <SectionsContext.Provider value={{ sections, refresh }}>
      {children}
    </SectionsContext.Provider>
  );
}

export const useSections = () => useContext(SectionsContext);

The refresh function is exposed so that when an admin creates or deletes a section, the page can call refresh() to re-fetch the sections list. Without this, the sidebar would still show the old list after a section is added.

Where Are Providers Placed — And Why?

Provider placement determines which components can consume the context. Look at App.jsx:

App() │ ├── AuthProvider ← OUTERMOST: whole app needs auth info │ └── BrowserRouter │ └── Routes │ ├── /login → LoginPage (uses useAuth to check if already logged in) │ │ │ └── ProtectedLayout (uses useAuth → checked BEFORE rendering) │ └── SectionsProvider ← INNER: only logged-in users need sections │ └── Layout │ ├── Sidebar (uses useSections for section nav links) │ └── Outlet → pages (use useSections for dropdowns)

SectionsProvider is placed inside ProtectedLayout for two reasons:

  1. Anonymous users don't need sections — saves an unnecessary API call
  2. The sections fetch runs only once the user is confirmed logged in

When Does Context Cause a Re-render?

When the value passed to a Provider changes, every component that consumes that context re-renders. So when setUser() is called in AuthContext, every component using useAuth() re-renders. This is usually fine since auth changes are rare events.

Context is not a replacement for all state. Use it for data that is truly global (user session, theme, language, shared lists). Use local useState for data that belongs to one component (form inputs, modal open/closed, loading flags).
