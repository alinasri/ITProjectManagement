🚀 Entry Point: main.jsx & App.jsx

The app starts with two files. main.jsx boots React into the browser. App.jsx defines the entire route structure and global providers. This lesson dissects both line by line.

Step 1 — The Browser Loads index.html

When you open the app in a browser, the server sends frontend/index.html. The relevant parts are:

<body>
  <div id="root"></div>               ← empty container
  <script type="module" src="/src/main.jsx"></script>
</body>

The browser downloads and runs main.jsx next.

Step 2 — main.jsx Boots React

// src/main.jsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App.jsx'

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>,
)

Let's break down each line:

LineWhat it does
import { StrictMode }A React helper that finds bugs by running effects twice in development
import { createRoot }React's DOM renderer — connects React to the browser DOM
import './index.css'Loads global CSS (Tailwind base styles)
import AppThe root React component that contains everything
createRoot(document.getElementById('root'))Finds the empty <div id="root"> in index.html
.render(<StrictMode><App /></StrictMode>)Mounts the entire React app into that div

StrictMode is only active in development. It intentionally runs useEffect and other side effects twice on mount to expose bugs that depend on side effects running exactly once. This is why AuthContext.jsx uses a useRef guard — to prevent the auth check from running twice.

Step 3 — App.jsx: The Root Component

App.jsx is the most important file in the frontend. It wires together the router, the global providers, and all the route definitions. Here's the full file with annotations:

// src/App.jsx
import { BrowserRouter, Routes, Route, Navigate, Outlet } from 'react-router-dom';
import { AuthProvider, useAuth } from './context/AuthContext';
import { SectionsProvider } from './context/SectionsContext';
import Layout from './components/Layout';
import LoginPage from './pages/LoginPage';
// ... more imports

The Helper Components Defined in App.jsx

App.jsx defines 5 small helper components before the main App component. Each solves a specific routing problem.

1. FullPageSpinner

function FullPageSpinner() {
  return (
    <div className="flex items-center justify-center h-screen bg-gray-950">
      <div className="w-8 h-8 border-2 border-indigo-500 border-t-transparent rounded-full animate-spin" />
    </div>
  );
}

A simple animated spinner shown while the app is checking whether the user is logged in. The Tailwind class animate-spin rotates the element. border-t-transparent makes only three sides of the border visible, creating the arc effect.

2. PublicOnly

function PublicOnly({ children }) {
  const { user, loading } = useAuth();
  // If already logged in (and auth check is done), redirect away from login
  if (!loading && user && !user.must_change_password) {
    return <Navigate to="/" replace />;
  }
  return children;
}

Wraps the Login page. If you're already logged in and navigate to /login, it redirects you home instead. Crucially, it does not block the login page while loading is true — the login form appears immediately. Only after the auth check finishes does it redirect if needed.

3. ProtectedLayout

function ProtectedLayout() {
  const { user, loading } = useAuth();
  if (loading) return <FullPageSpinner />;
  if (!user)   return <Navigate to="/login" replace />;
  if (user.must_change_password) return <Navigate to="/change-password" replace />;
  return (
    <SectionsProvider>
      <Layout />
    </SectionsProvider>
  );
}

This is the gate for all dashboard routes. It:

  1. Shows a spinner while checking auth status
  2. Redirects to /login if not authenticated
  3. Redirects to /change-password if password must be changed
  4. Otherwise renders the Layout (sidebar + content area) wrapped in SectionsProvider

Notice that SectionsProvider is inside ProtectedLayout — sections are only fetched once the user is confirmed to be logged in. No point fetching sections for anonymous visitors.

4. RoleRedirect

function RoleRedirect() {
  const { user, loading } = useAuth();
  if (loading) return <FullPageSpinner />;
  if (!user) return <Navigate to="/login" replace />;
  if (user.role === 'super_admin')   return <Navigate to="/admin" replace />;
  if (user.role === 'it_head')       return <Navigate to="/it-head" replace />;
  if (user.role === 'section_head')  return <Navigate to={`/section/${user.section_id}`} replace />;
  // ... other roles
}

Handles the root URL /. Instead of showing any content itself, it looks at the user's role and sends them to their home page. A super_admin goes to /admin, a section_head goes to their own section page, etc.

5. RequireAuth

function RequireAuth({ roles }) {
  const { user, loading } = useAuth();
  if (loading) return null;
  if (!user) return <Navigate to="/login" replace />;
  if (roles && !roles.includes(user.role)) return <Navigate to="/" replace />;
  return <Outlet />;
}

A role guard. If the optional roles prop is provided and the user's role is not in that list, they're redirected home. <Outlet /> renders the matched child route. This is what makes nested routes work — more on this in Lesson 04.

The Main App Component — Route Definitions

export default function App() {
  return (
    <AuthProvider>                         ← wraps EVERYTHING with auth state
      <BrowserRouter>                      ← enables client-side routing
        <Routes>

          // Public routes (no auth needed)
          <Route path="/login"
                 element={<PublicOnly><LoginPage /></PublicOnly>} />
          <Route path="/report/:token"
                 element={<PublicReport />} />

          // Change password — needs login but no role check
          <Route element={<RequireAuth />}>
            <Route path="/change-password" element={<ChangePasswordPage />} />
          </Route>

          // All dashboard routes — share one persistent Layout
          <Route element={<ProtectedLayout />}>
            <Route element={<RequireAuth roles={['super_admin']} />}>
              <Route path="/admin/*" element={<AdminDashboard />} />
            </Route>
            // ... more role-gated routes
          </Route>

          // Root URL — redirect based on role
          <Route path="/" element={<RoleRedirect />} />
          // Any unknown URL — redirect to home
          <Route path="*" element={<Navigate to="/" replace />} />

        </Routes>
      </BrowserRouter>
    </AuthProvider>
  );
}

Why is AuthProvider the Outermost Wrapper?

The nesting order matters. AuthProvider wraps BrowserRouter which wraps everything else. This means every component inside — including the router and all route components — can call useAuth() to get the current user. If you put AuthProvider inside BrowserRouter, route guards like RequireAuth (which use useAuth) would still work, but the login redirect in the 401 interceptor (in api/client.js) needs access to window.location, not the router — so placement doesn't matter for that.

Think of providers as context bubbles. Everything nested inside a Provider can access that Provider's data. The deeper you put a Provider, the fewer components can reach it. AuthProvider is the outermost because literally everything in the app might need to know who is logged in.
