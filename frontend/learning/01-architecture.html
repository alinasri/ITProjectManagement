🏗️ Architecture Overview

Before reading a single line of React code, you need a mental map of what React even is, what this app is built from, and how all the pieces connect to each other.

What Is React?

React is a JavaScript library for building user interfaces. It was created by Facebook in 2013. The core idea is simple: instead of manually manipulating HTML elements (like you would with jQuery), you describe what the UI should look like, and React figures out how to make the browser show it.

Imagine you have a counter showing the number 5. With plain JavaScript you would find the element and set its text: document.getElementById('count').textContent = '6'. With React, you just say "the counter is now 6" and React automatically updates every part of the page that depends on that number.

React is declarative. You describe what you want ("show a list of 5 items"). Plain JavaScript is imperative — you describe how to do it ("find the ul, create 5 li elements, append each one..."). Declarative code is much easier to read and maintain.

What Is a Single Page Application (SPA)?

Traditional websites load a new HTML page every time you click a link. Click "Purchases" → the server sends back an entirely new HTML file. The browser discards everything and starts over.

A Single Page Application loads one HTML file once, and then JavaScript swaps out only the content that changes. When you click "Purchases" in our sidebar, no new page loads from the server. React simply re-renders the main content area to show the purchases table. The URL in the browser changes (so bookmarks and the back button still work), but there is no page reload.

Traditional multi-page appSingle Page App (this project)
Server sends a new HTML page for every linkServer sends one HTML file, JS handles navigation
Full page reload on every clickInstant navigation — only content area changes
Browser discards all state on navigationState (user info, open modals) survives navigation
Sidebar re-renders every page loadSidebar stays mounted — no flicker

The Single HTML File

Open frontend/index.html. You'll find almost nothing there:

<div id="root"></div>
<script type="module" src="/src/main.jsx"></script>

That's it. An empty <div> and a script tag. React takes over that <div id="root"> and fills it with the entire application. Everything you see — the sidebar, the tables, the modals — is injected into that one div by JavaScript.

The Data Flow

Understanding how data moves through the app is the key to understanding React. Here is the general pattern for every page in this project:

User clicks something │ ▼ Component receives the event (onClick, onSubmit...) │ ▼ API function called (e.g. purchasesApi.list()) │ ← HTTP request sent to backend ▼ Backend processes request, returns JSON data │ ▼ State updated (setList(r.data)) │ ▼ React re-renders the component with new data │ ▼ User sees the updated UI

How the Frontend Connects to the Backend

The frontend runs on port 5173 (Vite's dev server). The backend runs on port 3000 (Express). When React code calls /api/purchases, Vite's dev server automatically forwards it to http://localhost:3000/api/purchases. This is configured in vite.config.js:

// vite.config.js
server: {
  proxy: {
    '/api': 'http://localhost:3000',
  },
}

So from the frontend's point of view, there's just one server. Vite handles the forwarding transparently in development. In production, a reverse proxy (like Nginx) would do the same job.

Pages vs Components — What's the Difference?

This is a naming convention, not a React rule. In this project:

  • Pages (src/pages/) — full-screen views that map 1-to-1 with a URL. /purchases → PurchasesPage.jsx
  • Components (src/components/) — smaller, reusable UI pieces that pages assemble together. Modal, StatCard, FilterBar etc.

Technically pages and components are identical — both are React components (functions that return JSX). The folder separation is just for organization so you can find things quickly.
