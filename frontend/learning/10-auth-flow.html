🔐 Authentication Flow

Authentication in this app is a cycle: check session on load → login form → protected routes → session expiry or logout. This lesson maps every step to the specific code that handles it.

The Big Picture

App loads in browser │ ▼ AuthProvider mounts user = null, loading = true │ ▼ useEffect fires → auth.me() call "Is there a valid session cookie?" │ ├── YES: setUser(userData), setLoading(false) │ → protected routes become accessible │ └── NO: setUser(null), setLoading(false) → ProtectedLayout redirects to /login

Phase 1 — App Boot: Session Check

The very first thing the app does when it loads is check if the user is already logged in. This happens inside AuthProvider:

// context/AuthContext.jsx
export function AuthProvider({ children }) {
  const [user, setUser]       = useState(null);   ← starts as null (unknown)
  const [loading, setLoading] = useState(true);   ← starts as true (checking)
  const checked = useRef(false);

  useEffect(() => {
    if (checked.current) return;  ← runs exactly once (StrictMode guard)
    checked.current = true;

    auth.me()                       ← GET /api/auth/me
      .then(r => setUser(r.data))  ← 200 OK: session exists
      .catch(() => setUser(null)) ← 401: no session
      .finally(() => setLoading(false));
  }, []);
  ...
}

The backend's GET /api/auth/me route reads the session cookie. If valid, it returns the user object ({ id, username, role, section_id, ... }). If not valid, it returns 401.

The loading flag prevents premature redirects

During the auth.me() request, loading is true and user is null. If ProtectedLayout only checked user:

// ❌ Without loading check — sends logged-in users to /login!
if (!user) return <Navigate to="/login" />;   ← fires immediately when user=null

// ✅ With loading check — waits for the check to finish
if (loading) return <FullPageSpinner />;     ← show spinner while checking
if (!user)   return <Navigate to="/login" />; ← only check after result is known

Phase 2 — The Login Flow

When loading finishes and user is still null, the user is on the login page. Here's what happens when they submit the form:

User types username + password, clicks "ورود" │ ▼ handleSubmit() called (e.preventDefault() stops page reload) │ ▼ setLoading(true) → button shows "در حال ورود..." and is disabled │ ▼ login(username, password) called (from useAuth) → auth.login({ username, password }) → POST /api/auth/login │ ├── 200 OK: { user: { id, role, ... } } │ → setUser(r.data.user) (in AuthContext) │ → return r.data.user (back to LoginPage) │ → navigate('/admin') (or role-specific route) │ └── 401 Error: { error: 'Invalid credentials' } → setError('Invalid credentials') → setLoading(false)

The login function in detail

// LoginPage.jsx — the form submit handler
const handleSubmit = async (e) => {
  e.preventDefault();
  setError('');
  setLoading(true);
  try {
    const user = await login(username, password);
    // Check if password change is required
    if (user.must_change_password) return navigate('/change-password');
    // Route by role
    if (user.role === 'super_admin') navigate('/admin');
    else if (user.role === 'it_head')  navigate('/it-head');
    else navigate(`/section/${user.section_id}`);
  } catch (err) {
    setError(err.response?.data?.error || 'خطا در ورود به سیستم');
  } finally {
    setLoading(false);
  }
};

What the backend does with a successful login

The backend verifies the credentials, creates a JWT, and sets it as an httpOnly cookie in the response. The browser automatically stores this cookie and sends it with every future request to the same domain. JavaScript cannot read this cookie — it's completely invisible to frontend code. This is a security feature: even if someone injects malicious JavaScript into your page, they can't steal the session token.

Phase 3 — Protected Routes

Once logged in, the user navigates within the app. Every protected page is behind the ProtectedLayout → RequireAuth guard chain:

// App.jsx — purchases route requires one of these roles
<Route element={<ProtectedLayout />}>
  <Route element={<RequireAuth roles={['super_admin', 'it_head', 'purchase_admin']} />}>
    <Route path="/purchases" element={<PurchasesPage />} />
  </Route>
</Route>

// RequireAuth checks role
function RequireAuth({ roles }) {
  const { user, loading } = useAuth();
  if (loading) return null;
  if (!user)   return <Navigate to="/login" replace />;
  if (roles && !roles.includes(user.role)) return <Navigate to="/" replace />;
  return <Outlet />;  ← render PurchasesPage
}

Frontend auth guards are UI, not security. A clever user could bypass RequireAuth in the browser's dev tools. The real security is the backend — every API endpoint also checks the JWT and role. The frontend guards exist purely for user experience (showing the right page to the right user, not flashing a forbidden page).

Phase 4 — Session Expiry (The 401 Interceptor)

JWT tokens expire. If a user keeps a tab open for hours and the token expires, the next API call will return 401. The interceptor in client.js handles this:

// api/client.js
api.interceptors.response.use(
  r => r,
  err => {
    const publicPaths = ['/login', '/change-password'];
    const onPublicPage = publicPaths.some(p => window.location.pathname.startsWith(p));
    if (err.response?.status === 401 && !onPublicPage) {
      window.location.href = '/login';  ← hard redirect (reloads app)
    }
    return Promise.reject(err);
  }
);

Using window.location.href instead of React Router's navigate() causes a full page reload. This is intentional — it clears all React state (including the stale user value) and starts fresh with a blank session.

Phase 5 — Logout

The user clicks "خروج" (logout) in the Sidebar:

// Sidebar.jsx
const handleLogout = async () => {
  await logout();            ← from useAuth()
  navigate('/login');
};

// AuthContext.jsx — logout function
const logout = async () => {
  await auth.logout();       ← POST /api/auth/logout → backend clears cookie
  setUser(null);             ← clear local state
};

Setting user to null in AuthContext causes every component that uses useAuth() to re-render. ProtectedLayout re-renders, sees user === null (and loading === false), and renders <Navigate to="/login" />. The user is redirected before navigate('/login') even runs — but having both is harmless.

The must_change_password Flow

Some users are created by the admin with a temporary password. The must_change_password flag in the user object triggers a mandatory password change before they can access any page:

// ProtectedLayout — checked on every render
if (user.must_change_password) return <Navigate to="/change-password" replace />;

// PublicOnly — also redirects away from login if must_change_password
if (!loading && user && !user.must_change_password) {
  return <Navigate to="/" replace />;
}
// If must_change_password IS set, don't redirect from login —
// they need to go to /change-password instead (handled by ProtectedLayout)

After the user changes their password, ChangePasswordPage calls setUser({ ...user, must_change_password: false }) (updating the context), and the next navigation takes them to their proper home page instead of the change-password page again.
