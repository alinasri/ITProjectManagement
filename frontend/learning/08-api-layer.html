🔌 API Layer

The frontend needs to talk to the backend. This project uses axios for HTTP requests, with a clean two-file structure: client.js for setup and index.js for all the API calls organized by domain.

What Is axios?

Axios is a JavaScript library for making HTTP requests — the same things a browser does when you navigate to a URL, but done programmatically in JavaScript. It's an alternative to the browser's built-in fetch() API with a nicer interface.

You give axios a URL and a method (GET, POST, PUT, etc.) and it sends the request and gives you back the response as a JavaScript object.

File 1: client.js — The axios Instance

// src/api/client.js
import axios from 'axios';

const api = axios.create({
  baseURL: '/api',              ← every request starts with /api
  withCredentials: true,       ← send cookies with every request
});

api.interceptors.response.use(
  r => r,      ← successful response: pass through unchanged
  err => {
    const publicPaths = ['/login', '/change-password'];
    const onPublicPage = publicPaths.some(p => window.location.pathname.startsWith(p));
    if (err.response?.status === 401 && !onPublicPage) {
      window.location.href = '/login';   ← session expired → force logout
    }
    return Promise.reject(err);
  }
);

export default api;

Let's unpack the three key configurations:

baseURL: '/api'

Every API call automatically gets /api prepended. So api.get('/purchases') actually sends GET /api/purchases. You never type /api in the rest of the code — it's centralized here.

Vite then proxies /api/* to http://localhost:3000/api/* (the backend). In production, a web server like Nginx does the same thing.

withCredentials: true

This tells axios to include cookies in every request. Without this, the session cookie (containing the JWT) would not be sent to the backend, and every request would appear unauthenticated.

The backend stores the JWT in an httpOnly cookie. This means JavaScript cannot read or modify the cookie — only the browser sends it automatically with each request. This is more secure than storing the token in localStorage (where any JavaScript on the page could steal it). But it means you MUST set withCredentials: true so axios sends the cookie.

The 401 Interceptor — Automatic Session Expiry Handling

An interceptor is a function that runs on every response before it reaches your code. The error interceptor here checks if any request got a 401 Unauthorized response:

  • If on a public page (login or change-password): do nothing — a 401 on the login page is normal (wrong password)
  • If on any other page: the session has expired → redirect to /login

This means you never have to handle session expiry in individual components. It's handled globally here. If a user leaves their browser open overnight and the session expires, the next time they click anything that calls the API, they'll be automatically redirected to login.

File 2: index.js — Organized API Functions

All API calls are defined in one place, organized by domain. This means pages never construct URLs manually — they call named functions:

// src/api/index.js
import api from './client';

export const auth = {
  login:          (data) => api.post('/auth/login', data),
  logout:         ()     => api.post('/auth/logout'),
  me:             ()     => api.get('/auth/me'),
  changePassword: (data) => api.post('/auth/change-password', data),
};

export const purchases = {
  list:       (params)        => api.get('/purchases', { params }),
  create:     (data)          => api.post('/purchases', data),
  update:     (id, data)      => api.put(`/purchases/${id}`, data),
  remove:     (id)            => api.delete(`/purchases/${id}`),
  getHistory: (id)            => api.get(`/purchases/${id}/history`),
  archive:    (id, archive)   => api.patch(`/purchases/${id}/archive`, { archive }),
};

Notice the HTTP method names:

axios methodHTTP verbUse for
api.get(url)GETFetching data
api.post(url, data)POSTCreating a new record
api.put(url, data)PUTReplacing/updating a record
api.patch(url, data)PATCHPartial update (e.g. toggle archive)
api.delete(url)DELETEDeleting a record

How Pages Use the API

Pages import only the domain objects they need, not the raw axios instance:

// PurchasesPage.jsx — imports
import { purchases as purchasesApi } from '../api';
// renamed to "purchasesApi" to avoid naming conflict with the state variable "purchases"

// Then use it:
const fetchAll = useCallback(async () => {
  const r = await purchasesApi.list(showArchived ? { archived: 1 } : {});
  setList(r.data);   ← r.data is the JSON response body from the backend
}, [showArchived]);

What does a response look like?

// axios wraps the response in an object:
r = {
  data:    [...],   ← THE ACTUAL RESPONSE BODY (what the backend sent)
  status:  200,     ← HTTP status code
  headers: {...},   ← response headers
  config:  {...},   ← request config
}

// So r.data is what you usually care about
const purchases = r.data;   ← the array of purchase objects

Handling errors

// Pattern used in LoginPage.jsx
try {
  const user = await login(username, password);
  navigate('/admin');
} catch (err) {
  // err.response.data is the JSON error body from the backend
  // e.g. { error: 'Invalid credentials' }
  setError(err.response?.data?.error || 'خطا در ورود به سیستم');
}

The ?.data?.error chain is optional chaining — if err.response is undefined (network error, no internet), it gracefully falls back to the default error message instead of crashing.

The query params pattern

Several API functions accept query parameters:

// api/index.js
list: (params) => api.get('/purchases', { params }),

// Usage
purchasesApi.list({ archived: 1 })
// Sends: GET /api/purchases?archived=1

purchasesApi.list({})
// Sends: GET /api/purchases   (no query params — empty object)

purchasesApi.list()
// Sends: GET /api/purchases   (no params at all)

Axios automatically serializes the params object into a URL query string. You never have to build ?key=value&key2=value2 strings manually.
