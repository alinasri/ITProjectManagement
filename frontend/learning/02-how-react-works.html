⚛️ How React Works

Components, JSX, props, and state — these are the four pillars of everything you see in this app. This lesson explains all four with real examples from the codebase.

Pillar 1 — Components

A component is just a JavaScript function that returns some HTML-like code. That's it. Here is the simplest possible React component:

function Greeting() {
  return <h1>Hello, world!</h1>;
}

React collects all these functions and composes them into a tree. Your entire app is a tree of nested component function calls, with App at the root.

App ├── AuthProvider ← context/AuthContext.jsx │ └── BrowserRouter ← React Router │ ├── LoginPage ← pages/LoginPage.jsx │ └── ProtectedLayout │ ├── Layout ← components/Layout.jsx │ │ ├── Sidebar ← components/Sidebar.jsx │ │ └── (page content via Outlet) │ │ ├── PurchasesPage │ │ │ ├── PageHeader │ │ │ ├── StatCard │ │ │ ├── FilterBar │ │ │ └── Modal │ │ └── ... other pages │ └── ... └── ...

When React renders the tree, it calls each function top-to-bottom. The output of all of them is combined into the final HTML that the browser displays.

Component names must start with a capital letter. LoginPage is a React component. loginPage would be interpreted as a plain HTML element (which doesn't exist) and React would silently ignore it.

Pillar 2 — JSX

The HTML-like syntax you see inside component functions is called JSX (JavaScript XML). It looks like HTML but it's actually JavaScript. Vite transforms it into regular JavaScript before the browser sees it.

For example, this JSX from LoginPage.jsx:

return (
  <div className="min-h-screen bg-gray-950">
    <h1 className="text-2xl font-bold text-white">
      سامانه مدیریت پروژه
    </h1>
  </div>
);

Gets transformed by Vite into something like:

return React.createElement('div', { className: 'min-h-screen bg-gray-950' },
  React.createElement('h1', { className: 'text-2xl font-bold text-white' },
    'سامانه مدیریت پروژه'
  )
);

Important JSX rules

  • Use className, not class — class is a reserved word in JavaScript, so JSX uses className for CSS classes.
  • Self-closing tags must close — <input /> not <input>
  • Only one root element — a component can only return one outermost element. Wrap multiple elements in a <div> or in a Fragment (<>...</>).
  • JavaScript expressions go in curly braces — {user.username}, {isLoading ? 'Loading...' : 'Submit'}

Curly braces — JavaScript inside JSX

Curly braces { } let you embed any JavaScript expression inside JSX. This is how dynamic content works:

// From Sidebar.jsx
<p className="text-sm text-white">{user?.username}</p>
<p className="text-xs text-gray-500">{roleLabel}</p>

// Show a spinner OR the content
{loading ? <Spinner /> : <PurchasesTable />}

// Render a list
{sections.map(s => (
  <SidebarLink key={s.id} to={`/section/${s.id}`} label={s.name} />
))}

The ?. in user?.username is called optional chaining. It means "if user is null or undefined, return undefined instead of crashing". Since user starts as null while the app is loading, this prevents errors.

Pillar 3 — Props

Props (short for "properties") are how a parent component passes data down to a child component. Think of them like function arguments — the parent calls the child and passes in values.

Here is StatCard — a component that accepts props:

// components/StatCard.jsx
export default function StatCard({ label, value, icon, color = 'indigo' }) {
  // label, value, icon, color are all PROPS — passed in by the parent
  return (
    <div className={`rounded-2xl border p-5 ${colors[color]}`}>
      <p className="text-3xl font-bold text-white">{value}</p>
      <p className="text-sm">{label}</p>
    </div>
  );
}

And here is how a page uses StatCard and passes props to it:

// Parent component (a dashboard page)
<StatCard
  label="تعداد خریدها"
  value={purchases.length}
  icon=<ShoppingCart />
  color="indigo"
/>

Props flow in one direction only: parent → child. A child component cannot change its props, and cannot send data back up to the parent through props. (For sending data up, you pass a callback function as a prop — more on that below.)

The children prop — special content slot

Modal is a great example. It accepts a children prop which is whatever you put between the opening and closing tags:

// Usage in a page — everything between <Modal> and </Modal> is "children"
<Modal open={showModal} onClose={() => setShowModal(false)} title="Add Purchase">
  <form>
    <input type="text" placeholder="Title" />
    <button>Save</button>
  </form>
</Modal>

// Inside Modal.jsx, the "children" prop renders that form
<div className="px-6 py-5">{children}</div>

Callback props — sending data up

Since props only flow downward, how does a child tell its parent something happened? The parent passes a function as a prop, and the child calls it:

// Parent passes a function as prop
<FilterBar
  searchTerm={searchTerm}
  onSearchChange={setSearchTerm}  ← function passed as prop
/>

// FilterBar.jsx calls it when the user types
<input
  value={searchTerm}
  onChange={e => onSearchChange(e.target.value)}  ← calls parent's function
/>

When the user types in the search box, FilterBar calls onSearchChange with the new value. This actually runs setSearchTerm in the parent, updating the parent's state, which causes both the parent and FilterBar to re-render with the new search term.

Pillar 4 — State

State is data that belongs to a component and can change over time. When state changes, React automatically re-renders the component to show the updated UI. This is the entire point of React — you update state, React updates the screen.

State is created with the useState hook:

import { useState } from 'react';

// From LoginPage.jsx
const [username, setUsername] = useState('');
const [password, setPassword] = useState('');
const [error, setError]       = useState('');
const [loading, setLoading]   = useState(false);

useState('') returns an array with two elements:

  1. The current value of the state (username)
  2. A setter function to update it (setUsername)

The argument to useState() is the initial value. Here username starts as an empty string.

You MUST use the setter — never mutate directly

// ❌ WRONG — React won't know the state changed, won't re-render
username = 'ali';

// ✅ CORRECT — React sees the change and re-renders
setUsername('ali');

State causes re-renders

Whenever you call a setter, React schedules a re-render of that component (and its children). The component function runs again, JSX is computed again, and React updates only the parts of the DOM that changed.

// From LoginPage.jsx — when user types, state updates, input re-renders
<input
  type="text"
  value={username}               ← controlled by state
  onChange={e => setUsername(e.target.value)}  ← updates state on every keystroke
/>

This is called a controlled input. The input's value is always driven by React state. Typing in the box calls setUsername, which updates state, which re-renders the input with the new value. React is always the source of truth for what's in the box.

Re-rendering — The Mental Model

Understanding re-rendering is the key to understanding React bugs. Here is the simplified mental model:

  1. State changes (you call a setter)
  2. React calls your component function again from the top
  3. All variables inside the function are recalculated
  4. New JSX is returned
  5. React compares the new JSX to the previous JSX (this is called "diffing")
  6. Only the actual DOM elements that differ are updated

Because the component function runs again from top to bottom on every render, local variables (non-state) are reset every render. Only useState, useRef, and context values survive re-renders. This is why state exists — it's the only way to remember things across renders.

Putting It Together — LoginPage.jsx

Let's read LoginPage through the lens of what you just learned:

export default function LoginPage() {
  // STATE — data that changes and drives the UI
  const [username, setUsername] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError]       = useState('');
  const [loading, setLoading]   = useState(false);

  // HOOK — useAuth() gives us the login function from AuthContext
  const { login } = useAuth();
  const navigate = useNavigate();

  // EVENT HANDLER — called when form is submitted
  const handleSubmit = async (e) => {
    e.preventDefault();   // stop browser from reloading the page
    setError('');          // clear previous error
    setLoading(true);      // show spinner on button
    try {
      const user = await login(username, password);  // call API
      navigate('/admin');   // redirect on success
    } catch (err) {
      setError(err.response?.data?.error || 'خطا در ورود');
    } finally {
      setLoading(false);    // always hide spinner
    }
  };

  // JSX — describes what to render based on current state
  return (
    <form onSubmit={handleSubmit}>
      {error && <div className="text-red-300">{error}</div>}
      <input value={username} onChange={e => setUsername(e.target.value)} />
      <button disabled={loading}>
        {loading ? 'در حال ورود...' : 'ورود'}
      </button>
    </form>
  );
}

Notice how the JSX reacts to state: {error && ...} shows the error only when error is non-empty. {loading ? '...' : 'ورود'} switches the button text. disabled={loading} disables the button while the request is in flight. All of this happens automatically — you just update state and React handles the DOM.
