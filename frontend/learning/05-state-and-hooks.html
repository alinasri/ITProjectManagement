🪝 State & Hooks

Hooks are functions that let you use React features inside functional components. This lesson covers the four hooks used throughout this project: useState, useEffect, useRef, and useCallback.

What Are Hooks?

Before hooks (React 16.8, 2019), you could only use React features like state in "class components" — a verbose, complex JavaScript class syntax. Hooks let you use these same features in simple functions. Today, no one writes class components for new code.

All hook names start with use. React provides built-in hooks, and you can also write your own "custom hooks" (like useAuth and useSections in this project).

Rules of Hooks — you MUST follow these or React will break:
1. Only call hooks at the top level of a component (not inside if/else, loops, or nested functions).
2. Only call hooks from React function components (or other custom hooks).
ESLint will warn you if you break either rule.

useState — Local Component State

You've already seen useState in Lesson 02. Here's a deeper look at how it's used across this project.

Basic pattern

const [value, setValue] = useState(initialValue);
  • value — the current state value
  • setValue — the function to update it (triggers a re-render)
  • initialValue — what the state is on first render

State shapes used in PurchasesPage

// Primitives
const [loading, setLoading]           = useState(true);
const [showArchived, setShowArchived] = useState(false);
const [searchTerm, setSearchTerm]     = useState('');
const [editingId, setEditingId]       = useState(null);

// Array of objects (the actual data)
const [list, setList]         = useState([]);

// Object (form data for the row being edited)
const [editRow, setEditRow]   = useState({});

Updating object state

When state is an object, you must spread the old values and override only what changed. You cannot partially update state — each call to the setter replaces the whole thing:

// ❌ WRONG — this replaces the whole editRow with just { status: 'done' }
setEditRow({ status: 'done' });

// ✅ CORRECT — spread old values, override status
setEditRow(prev => ({ ...prev, status: 'done' }));

The functional update form prev => ({ ...prev, ... }) is used when the new value depends on the old value. It guarantees you're working with the most recent state, even if multiple updates are batched.

useEffect — Side Effects

A "side effect" is anything that reaches outside the component's render logic: fetching data from an API, setting up a timer, adding event listeners, etc. useEffect is where you put this code so it runs at the right time.

useEffect(() => {
  // effect body — runs after render
  return () => {
    // optional cleanup — runs before the next effect or when component unmounts
  };
}, [/* dependency array */]);

The dependency array — the most important concept in useEffect

The second argument to useEffect is an array that controls when the effect runs:

Dependency arrayWhen the effect runs
No array (omitted)After EVERY render — almost never what you want
[] empty arrayOnce — only after the first render (component mounts)
[value] with valuesAfter the first render, and again whenever value changes

Example 1: fetch on mount (empty array)

// AuthContext.jsx — run auth.me() once when the provider first mounts
useEffect(() => {
  if (checked.current) return;  // guard against StrictMode double-fire
  checked.current = true;

  auth.me()
    .then(r => setUser(r.data))
    .catch(() => setUser(null))
    .finally(() => setLoading(false));
}, []);  ← empty array = runs once on mount only

Example 2: re-fetch when a value changes

// PurchasesPage.jsx — re-fetch when showArchived toggle changes
const fetchAll = useCallback(async () => {
  const r = await purchasesApi.list(showArchived ? { archived: 1 } : {});
  setList(r.data);
}, [showArchived]);

useEffect(() => { fetchAll(); }, [fetchAll]);
// fetchAll changes whenever showArchived changes (via useCallback)
// → this effect re-runs → API is called with the new filter

Example 3: effect with cleanup (event listener)

// Modal.jsx — add Escape key listener when modal opens, remove it when it closes
useEffect(() => {
  if (!open) return;                          ← skip if modal is closed
  const handler = (e) => {
    if (e.key === 'Escape') onClose();
  };
  document.addEventListener('keydown', handler);
  return () => document.removeEventListener('keydown', handler);  ← cleanup!
}, [open, onClose]);  ← re-run when open or onClose changes

The cleanup function is critical here. Without it, every time the modal opens, a new event listener is added. Eventually you'd have dozens of listeners all trying to close the modal. The cleanup removes the old listener before the next effect runs (or when the component unmounts).

Every value from the component scope that you use inside a useEffect should be listed in the dependency array. ESLint's react-hooks/exhaustive-deps rule enforces this. Forgetting a dependency leads to "stale closure" bugs where the effect uses an old value.

useRef — A Value That Survives Re-renders Without Causing Them

A ref is like a box you can put a value in. The value persists across renders (like state), but changing it does NOT cause a re-render (unlike state). It's used for:

  • Tracking something across renders that doesn't affect the UI
  • Holding a reference to a DOM element
const myRef = useRef(initialValue);
// Read the current value:
myRef.current  ← always the latest value
// Write to it (does NOT trigger a re-render):
myRef.current = newValue;

The StrictMode deduplication in AuthContext

This is a real example of why useRef is needed:

// context/AuthContext.jsx
const checked = useRef(false);

useEffect(() => {
  // React StrictMode in dev fires useEffect TWICE on mount
  // This guard ensures auth.me() is only called ONCE
  if (checked.current) return;
  checked.current = true;

  auth.me().then(r => setUser(r.data))...
}, []);

Why use a ref instead of state here? Because we don't want a re-render when we set checked = true. We just need a flag that persists between the two StrictMode calls. A state variable would cause a re-render.

useCallback — A Stable Function Reference

useCallback returns a memoized version of a function — the same function object is returned on every render, unless one of its dependencies changes. This matters because functions in JavaScript are recreated on every render.

Why does this matter? Look at PurchasesPage:

// Without useCallback — new function object every render
const fetchAll = async () => {
  const r = await purchasesApi.list(...);
  setList(r.data);
};
useEffect(() => { fetchAll(); }, [fetchAll]);
// Problem: fetchAll is a new function on every render
// → useEffect dependency changes every render → infinite loop!
// With useCallback — stable function reference
const fetchAll = useCallback(async () => {
  const r = await purchasesApi.list(showArchived ? { archived: 1 } : {});
  setList(r.data);
}, [showArchived]);  ← only recreate when showArchived changes

useEffect(() => { fetchAll(); }, [fetchAll]);
// Now: fetchAll only changes when showArchived changes
// → useEffect only re-runs when showArchived changes → correct!

useCallback(fn, deps) is the same as useMemo(() => fn, deps). They both memoize values — useMemo for any value, useCallback as a shortcut for memoizing functions. Don't use either blindly — they add overhead. Use them when a function is a dependency in a useEffect, or when passing stable callbacks to heavily optimized child components.

Custom Hooks — useAuth and useSections

A custom hook is just a function that starts with "use" and calls other hooks inside. They let you extract and reuse stateful logic.

// At the bottom of AuthContext.jsx
export const useAuth = () => useContext(AuthContext);

// This tiny custom hook wraps useContext so components just write:
const { user, login, logout } = useAuth();
// instead of the more verbose:
const { user, login, logout } = useContext(AuthContext);

Similarly for sections:

// At the bottom of SectionsContext.jsx
export const useSections = () => useContext(SectionsContext);

// Usage in any component
const { sections, refresh } = useSections();
