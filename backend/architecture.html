Backend Architecture Guide

A complete walkthrough of the Node.js + Express backend for developers new to the stack.

Core Concepts

Before reading individual files, these are the five ideas that underpin everything in an Express backend.

1. Node.js is the Runtime

Node.js lets you run JavaScript outside a browser. It handles the TCP socket, file I/O, and OS interactions. You write the application logic; Node handles everything underneath.

2. Express is the Framework

Express is a thin layer on top of Node that provides a structured way to define what happens when an HTTP request arrives. Without Express you would manually parse URLs, headers, and bodies yourself.

3. Middleware — the Pipeline

Every incoming request passes through a pipeline of functions before it gets a response. Each function has the signature (req, res, next). It can either respond immediately or call next() to hand the request to the next function in the chain.

// A simple middleware example
function logger(req, res, next) {
  console.log(req.method, req.url);   // do something
  next();                             // pass to the next function
}

// A middleware that stops the chain (guards a route)
function requireAuth(req, res, next) {
  if (!isLoggedIn) return res.status(401).json({ error: 'Unauthorized' });
  next(); // only reached if logged in
}

4. Routes — matching URLs to handlers

A route handler runs when an HTTP request matches a specific method (GET, POST…) and URL pattern. Route parameters like :id are extracted into req.params.id. Query strings like ?section_id=3 become req.query.section_id.

5. req and res — the request/response objects

Property/MethodWhat it contains
req.bodyParsed JSON body of POST/PUT requests (requires express.json())
req.params.idURL segment captured by /:id
req.query.xQuery string value from ?x=value
req.cookies.tokenCookie named "token" (requires cookie-parser)

res.json(data)Sends a JSON response and ends the request
res.status(404).json(...)Sets the HTTP status code before sending

Request Lifecycle

Here is the exact path a request like PUT /api/projects/7 takes through the system:

Browser → Node TCP → Express
  → cors()                   app.js: CORS headers
  → express.json()           app.js: parse body into req.body
  → cookieParser()           app.js: parse cookies into req.cookies
  → app.use('/api/projects') app.js: routes to projects router
  → requireAuth              middleware/auth.js: verify JWT
  → requireRole(...)         middleware/auth.js: check permission
  → router.put('/:id', ...)  routes/projects.js: do the work
  → res.json(updatedProject) response sent back to browser

server.js — Entry Point

backend/src/server.js

This is the file Node actually executes (node src/server.js). Its only job is to start the HTTP server. It is intentionally kept thin so that tests can import app.js without binding a real network port.

app.listen(PORT, callback)

Tells Node to start accepting TCP connections on the given port. The callback fires once — right after the server is ready. process.env.PORT reads a system environment variable; if it is not set, the fallback is 3000. This lets cloud hosts assign a port without changing code.

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Server running on port ${PORT}`));

app.js — Application Setup

backend/src/app.js

Creates the Express application object and wires it together: global middleware first, then route groups, then the production static file fallback.

express()

Creates the main app object. Everything — middleware and routes — is attached to this object.

app.use(cors(...))

CORS (Cross-Origin Resource Sharing) adds HTTP headers that tell browsers it is safe to send requests from a different origin. The React frontend (port 5173) and this API (port 3000) are different origins. Without CORS, the browser's security policy blocks all cross-origin requests. credentials: true is required so cookies are included in cross-origin calls.

app.use(express.json())

Reads the raw request body, parses it as JSON, and attaches the result to req.body. Without this, req.body is undefined in POST and PUT handlers.

app.use(cookieParser())

Parses the Cookie HTTP header and populates req.cookies. Required so requireAuth can read req.cookies.token.

app.use('/api/projects', require('./routes/projects'))

Route mounting: any request whose URL starts with /api/projects is delegated to the projects router. The match is prefix-based — /api/projects/5/history still goes to the projects router. Each domain (auth, users, projects…) lives in its own file.

Production static file serving

In production the React app is compiled into static HTML/JS/CSS files placed in backend/public/. express.static() serves them. The wildcard GET * catch-all returns index.html for any URL that didn't match an API route, letting React's client-side router handle navigation (so refreshing /projects/5 in the browser still works).

db/schema.js — Database, Schema & Migrations

backend/src/db/schema.js

This file runs once when the application starts and exports the database connection used everywhere else. It does three things in order: opens the database, creates tables if they don't exist, runs any pending migrations, and seeds initial data if the database is empty.

Opening the database

const db = new DatabaseSync(path.join(DATA_DIR, 'it_pm.db'));
db.exec("PRAGMA journal_mode = WAL");
db.exec("PRAGMA foreign_keys = ON");

Uses Node's built-in SQLite (node:sqlite). SQLite stores the entire database in a single .db file — no separate database server is needed. WAL mode improves concurrent read performance. foreign_keys = ON enforces the relationships between tables (e.g. a project cannot reference a non-existent section).

CREATE TABLE IF NOT EXISTS — Schema definition

Runs on every startup but is safe to repeat — the IF NOT EXISTS clause means it only creates the table if it isn't already there. This is the complete current schema for brand-new databases.

hasRun(id), stamp(id), runMigration(id, fn)

The migration system. A migration is a one-time database change (adding a column, renaming a table). The _migrations table records which migrations have already run. runMigration checks if a migration has run, executes it if not, then stamps it so it is never run again. This is how the database schema evolves over time without manually editing it.

function runMigration(id, fn) {
  if (hasRun(id)) return;   // already applied, skip
  fn();                      // apply the change
  stamp(id);                 // record it so it never runs again
}

Seed data

If the database is brand new (no sections exist), five default sections and one super_admin user (admin / admin123) are inserted. The password is run through bcrypt — it is never stored in plain text.

module.exports = db

Exports the single database connection. Every route file that does require('../db/schema') receives this same object, so there is one shared connection to the database throughout the application.

db/helpers.js — Shared Database Utilities

backend/src/db/helpers.js

Reusable database operations shared by multiple route files. Keeping these here avoids copy-pasting the same logic across five different route files.

recordStatusChange(entityType, entityId, fromStatus, toStatus, userId)

Writes an audit trail row to status_history whenever an entity's status changes. Silently skips the insert if fromStatus === toStatus — no point recording a no-op. The ?? (nullish coalescing) ensures null is stored instead of undefined, which SQLite does not understand.

recordFieldChange(entityType, entityId, field, fromValue, toValue, userId)

Like recordStatusChange but for non-status fields like due_date. Reuses the same status_history table; the field column distinguishes these rows from status changes.

setResponsibles(table, fk, id, personnelIds)

Replaces the complete list of people responsible for a project or task. The strategy is: delete all existing rows for this entity, then re-insert the new list. This "delete-and-re-insert" is simpler than computing a diff between the old and new lists, and handles additions and removals in one operation.

setResponsibles('project_responsibles', 'project_id', 7, [2, 5]);
// Result: project 7 is now assigned to personnel 2 and 5, nothing else.

setSections(table, fkCol, id, sectionIds)

Same delete-and-re-insert pattern for linking purchases, tenders, or contracts to multiple sections.

middleware/auth.js — Authentication & Authorization

backend/src/middleware/auth.js

Provides two middleware functions used across every protected route. Middleware always has the signature (req, res, next) — it either responds (ending the chain) or calls next() to continue.

What is a JWT? A JSON Web Token is a signed string that encodes a payload (user id, role, etc.). The server signs it with a secret. Anyone can decode the payload, but only the server can verify the signature. The client stores the token and sends it with every request — the server never needs to look it up in a database to know who you are (except for the active/deleted check).

requireAuth(req, res, next)

Verifies the caller has a valid, unexpired JWT. Looks for the token in two places: first the HTTP-only cookie (req.cookies?.token), then the Authorization header. HTTP-only cookies cannot be read by browser JavaScript, which protects against XSS attacks.

On success, attaches the decoded user to req.user so downstream handlers know who is making the request. Also re-queries the database to check if the user was disabled after the token was issued — without this, a deactivated account would remain usable until the token expires.

// Optional chaining (?.) prevents a crash if the property is undefined
const token = req.cookies?.token || req.headers.authorization?.split(' ')[1];
if (!token) return res.status(401).json({ error: 'Unauthorized' });
req.user = jwt.verify(token, JWT_SECRET); // throws if invalid/expired
next();

requireRole(...roles)

A higher-order function: it accepts a list of allowed roles and returns a middleware function. The returned middleware reads req.user.role (set by requireAuth) and rejects the request with 403 Forbidden if the role is not in the allowed list.

// Higher-order function — takes roles, returns middleware
function requireRole(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user?.role))
      return res.status(403).json({ error: 'Forbidden' });
    next();
  };
}

// Usage: three functions run in order
router.get('/', requireAuth, requireRole('super_admin'), handler);

routes/auth.js — Authentication Routes

backend/src/routes/auth.js

express.Router() creates a mini-app. It is mounted at /api/auth in app.js, so router.post('/login') handles POST /api/auth/login.

POST /login

Validates credentials, creates a JWT, and sends it as both a cookie and in the JSON body.

  • db.prepare('SELECT * FROM users WHERE username = ?').get(username) — the ? placeholder prevents SQL injection. Never concatenate user input into SQL strings.
  • bcrypt.compareSync(password, user.password_hash) — passwords are never stored in plain text. bcrypt hashes them one-way. This function compares a plain text password against the stored hash.
  • jwt.sign({ id, username, role, section_id }, JWT_SECRET, { expiresIn: '8h' }) — creates a token with an 8-hour expiry.
  • res.cookie('token', token, { httpOnly: true }) — stores the token in a cookie the browser automatically sends with every subsequent request. httpOnly: true means JavaScript cannot access it, protecting against XSS.

POST /logout

res.clearCookie('token') removes the token cookie from the browser. Simple and stateless.

GET /me — requires auth

Returns the currently logged-in user's profile. Because requireAuth already decoded the token into req.user, this just does a fresh database lookup to return current data.

POST /change-password — requires auth

Verifies the current password, then hashes and stores the new one. Also sets must_change_password = 0, clearing the "force change on first login" flag.

routes/users.js — User Management

backend/src/routes/users.js

All routes require requireAuth + requireRole('super_admin'). Only the super admin can manage users.

GET /

Returns all non-soft-deleted users (is_deleted = 0).

POST /

Creates a new user. super_admin cannot be created through this endpoint (only the initial seed creates one). New users get must_change_password = 1 so they set their own password on first login. result.lastInsertRowid is the auto-generated ID of the new row.

PUT /:id

Updates a user. req.params.id is extracted from the URL pattern /:id. Uses || and ?? to implement partial updates — only fields that are provided in the request body are changed.

PATCH /:id/toggle-active

Flips is_active between 0 and 1. Using PATCH (vs PUT) signals a partial update — HTTP convention for changing a single field.

DELETE /:id

Soft-delete: sets is_deleted = 1 rather than removing the row. The 10-minute age check prevents accidental deletion of established users — only freshly created accounts can be deleted this way.

routes/sections.js — Sections

backend/src/routes/sections.js

GET / — any authenticated user

Returns all sections. No role restriction — everyone needs to list sections.

POST / and PUT /:id — super_admin only

Standard create and update. Input is validated and trimmed before insertion.

DELETE /:id — super_admin only

Hard delete (the row is actually removed). Before deleting, it counts projects, tasks, and personnel in the section. If any exist, returns 409 Conflict with { error: 'has_data' } — the frontend shows a message explaining why the section cannot be deleted.

routes/personnel.js — Personnel

backend/src/routes/personnel.js

People who can be assigned as responsibles on projects and ongoing tasks.

GET / — role-based scoping inside the handler

A section_head always sees only their own section's personnel. The scoping is inside the handler (not in middleware) because the filtering rule is data-dependent, not just role-dependent. Admins can filter via ?section_id=3 (read from req.query) or get everyone.

POST /

A section_head's section is forced to their own req.user.section_id regardless of what the request body says — preventing them from creating personnel for other sections.

DELETE /:id

Hard delete with a section ownership check. Because of ON DELETE CASCADE in the schema, deleting a person automatically removes their entries from project_responsibles and ongoing_task_responsibles.

routes/projects.js — Projects

backend/src/routes/projects.js

The most complex route file — the core of the application. Worth understanding in detail.

enrichProjects(projects)

A local helper that takes raw project rows from the database and adds two computed fields to each: responsibles (the assigned people) and custom_values (values for custom columns). Called after every query so the frontend always receives a complete, consistent object. The spread { ...p, responsibles, custom_values } creates a new object combining all of p's fields with the two new arrays.

GET /

Three query branches depending on role. A section_head always sees only their own section. Others can filter via ?section_id= or get everything. ?archived=1 switches to showing archived items. The result is always passed through enrichProjects().

POST /

Creates a project. After the INSERT, result.lastInsertRowid gives the new row's ID. Calls recordStatusChange with fromStatus = null (the project didn't exist before). Then calls setResponsibles if responsible IDs were provided.

GET /deadline-changes

This route is defined before GET /:id/history in the file. If it were after, Express would match the string "deadline-changes" as the :id parameter. More specific routes must always come before wildcard routes.

Returns all due_date changes in the last 30 days, joining three tables to include the project title and the username of who made the change.

GET /:id/history

Returns all status and field change rows for a specific project from status_history.

PUT /:id

Full update. Uses ?? (nullish coalescing) throughout: title ?? project.title means "use the new title if provided, otherwise keep the old one". Handles three concerns: the main UPDATE, a status change audit entry, a due date change audit entry, responsible person updates, and custom column value upserts.

Custom values use an upsert: INSERT ... ON CONFLICT DO UPDATE SET — insert the row if it doesn't exist, update it if it does. No need to check first.

PATCH /:id/archive

Toggles is_archived. The check req.body.archive === false uses strict equality — only the boolean false unarchives; any other value (including missing the field) archives.

DELETE /:id

Soft delete. Sets is_deleted = 1; the row stays in the database. The 10-minute rule: only freshly created projects can be deleted, preventing accidental data loss on established records.

routes/ongoingTasks.js — Ongoing Tasks

backend/src/routes/ongoingTasks.js

Structurally identical to projects.js but simpler — no custom columns or due dates. enrichTasks() only attaches responsibles. All the same route patterns apply. Good file to read after projects to see the patterns repeated.

routes/purchases.js — Purchases

backend/src/routes/purchases.js

Unlike projects (one section per project), a purchase can belong to multiple sections via the purchase_sections join table.

enrichPurchases(rows)

Attaches the sections array by joining through purchase_sections. Every row gets the full list of sections it belongs to.

Section filtering with a JOIN

When filtering by section, a JOIN query is used instead of a simple WHERE: SELECT DISTINCT p.* FROM purchases p JOIN purchase_sections ps ON ps.purchase_id = p.id WHERE ps.section_id = ?. The DISTINCT prevents duplicate rows if a purchase is linked to multiple sections that all match the filter.

routes/tenders.js — Tenders

backend/src/routes/tenders.js

Same multi-section pattern as purchases. Tender-specific fields: estimated_amount, deadline, winner. Statuses: open → evaluating → awarded → completed or cancelled. The tender_admin role mirrors purchase_admin.

routes/contracts.js — Contracts

backend/src/routes/contracts.js

Same multi-section pattern. Contract-specific fields: counterparty, start_date, end_date. Statuses: active → renewed → expired or cancelled. The contract_admin role owns create/edit.

routes/customColumns.js — Custom Columns

backend/src/routes/customColumns.js

Allows sections to define their own extra columns that appear on their projects.

GET /

Section heads automatically see only their section's columns. Admins can filter via ?section_id= or see all.

POST /

Auto-increments column_order using COALESCE(MAX(column_order), 0) + 1. COALESCE returns the first non-null argument — if no columns exist yet (MAX returns NULL), it falls back to 0, making the first column order = 1.

DELETE /:id

Hard delete. Because custom_values has ON DELETE CASCADE in the schema, deleting a column automatically removes all stored values for that column across all projects.

routes/report.js — Public Report

backend/src/routes/report.js

A shareable, read-only snapshot of all live data, intended for stakeholders who don't have a login.

getOrCreateToken()

Lazily creates a random token if one doesn't already exist. crypto.randomBytes(24).toString('hex') generates a cryptographically secure 48-character hex string — effectively unguessable.

GET /token — super_admin or it_head

Returns the current shareable token to admins so they can construct and share the public report URL.

POST /regenerate — super_admin only

Generates a new token, which invalidates all previously shared links. Uses INSERT OR REPLACE (SQLite upsert) to overwrite the single row in report_tokens.

GET /public/:token — no authentication

The public endpoint. Anyone who knows the token can access it — the token in the URL acts as the authentication. The token is validated by comparing it against the stored one. Returns a full snapshot: sections, projects, tasks, purchases, tenders, and contracts.

This route intentionally has no requireAuth. That is correct design — it is meant to be publicly accessible via a shared link.

Design Patterns Used Throughout

PatternWhereWhy
Soft delete projects, tasks, purchases, tenders, contracts, users Setting is_deleted = 1 preserves history and audit trails. A hard DELETE removes the row forever, which could break foreign key references in status_history.
Prepared statements Every database call db.prepare('... WHERE id = ?').get(id) separates SQL structure from data values, preventing SQL injection attacks.
enrich* helper functions projects, ongoingTasks, purchases, tenders, contracts Attaches related data (responsibles, sections) in one place, so every route returns a consistent shape.
Delete-and-re-insert for relationships setResponsibles, setSections Simpler than diffing old vs new lists. Works for any combination of additions and removals in one operation.
Role-based data scoping inside handlers GET routes in projects, personnel, purchases… section_head users automatically see only their section's data, enforced server-side. The frontend cannot bypass this.
10-minute delete window users, projects, tasks, purchases, tenders, contracts Prevents accidental deletion of established records while still allowing immediate cleanup of just-created test entries.
module.exports / require Every file Node's module system. Each file is isolated; you explicitly export what you want to share and import what you need.
