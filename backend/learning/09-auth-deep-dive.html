🔐 Authentication Deep Dive

How do we know who someone is? How do we store passwords safely? How does the server remember a logged-in user across multiple requests? This lesson explains every piece of the authentication system.

The Problem: HTTP Is Stateless

HTTP doesn't "remember" anything. Each request is independent. If you make 100 requests, the server has no idea they came from the same person. We need a mechanism to prove identity on every request.

❌ Without auth: Browser → GET /api/projects Server: "Who are you? I don't know." ✅ With JWT in cookie: Browser → GET /api/projects Cookie: token=eyJ... ← "It's me, user 5, section_head" Server: "Token checks out. Welcome, Ali."

Step 1: Storing Passwords Safely

Passwords must never be stored as plain text. If the database leaks, attackers would have everyone's password. Instead, passwords are hashed using bcryptjs.

What is a hash?

A hash is a one-way transformation. You can turn "mypassword" into a hash, but you cannot turn the hash back into "mypassword". To verify, you hash the input again and compare the hashes.

const bcrypt = require('bcryptjs');

// When creating a user (or changing password):
const hash = bcrypt.hashSync('mypassword123', 10);
// hash = "$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
// The "10" is the cost factor: how much computational work to do
// Higher = slower = harder for attackers to brute-force

// To verify at login:
bcrypt.compareSync('mypassword123', hash);  // → true
bcrypt.compareSync('wrongpassword', hash);  // → false

bcrypt automatically generates a random salt (a unique random string appended to the password before hashing). This means two users with the same password get completely different hashes. An attacker can't use a precomputed "rainbow table" to crack bcrypt hashes.

Step 2: The Login Flow

File: src/services/authService.js

const bcrypt = require('bcryptjs');
const jwt = require('jsonwebtoken');
const userRepo = require('../repositories/userRepository');
const ServiceError = require('../utils/serviceError');

const JWT_SECRET = process.env.JWT_SECRET || 'change-this-secret-in-production';

function login({ username, password }) {
  if (!username || !password) throw new ServiceError(400, 'Missing credentials');

  const user = userRepo.findByUsername(username);

  // SECURITY: Never say "user not found" vs "wrong password" separately.
  // If we returned early when user is null, an attacker could enumerate
  // which usernames exist. We always run bcrypt.compareSync to take the
  // same amount of time regardless, then give the same error message.
  if (!user || !bcrypt.compareSync(password, user.password_hash)) {
    throw new ServiceError(401, 'Invalid username or password');
  }

  if (user.is_active === 0) throw new ServiceError(403, 'Account disabled');

  // Create a signed token containing the user's identity
  // This token is sent to the client and included in all future requests
  const token = jwt.sign(
    { id: user.id, username: user.username, role: user.role, section_id: user.section_id },
    JWT_SECRET,
    { expiresIn: '8h' }  // token expires after 8 hours
  );

  return {
    token,
    user: {
      id: user.id, username: user.username, role: user.role,
      section_id: user.section_id,
      must_change_password: user.must_change_password === 1,
    },
  };
}

Step 3: What Is a JWT?

JWT stands for JSON Web Token. It has three parts separated by dots:

eyJhbGciOiJIUzI1NiJ9 . eyJpZCI6NSwidXNlcm5hbWUiOiJhbGkifQ . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c ↑ ↑ ↑ Header Payload Signature (algorithm used) (the data — base64) (HMAC of header+payload using JWT_SECRET)

The payload is just base64-encoded JSON — anyone can decode it and read it. It is NOT encrypted. But it cannot be forged without the secret key, because the signature would not match.

// Payload (readable by anyone who has the token):
{
  "id": 5,
  "username": "ali",
  "role": "section_head",
  "section_id": 2,
  "iat": 1703000000,  // issued at (timestamp)
  "exp": 1703028800   // expires at (8 hours later)
}

// jwt.verify() checks:
// 1. Is the signature valid? (was this issued by our server?)
// 2. Is exp in the future? (is the token still valid?)
// If both pass, it returns the payload. If not, it throws.
req.user = jwt.verify(token, JWT_SECRET);

Never put sensitive data in a JWT payload — the password hash, full personal info, or anything you don't want users to see. The payload is readable by anyone who has the token.

Step 4: Storing the Token in a Cookie

File: src/routes/auth.js

// After login, the token is set as an HTTP cookie:
res.cookie('token', token, {
  httpOnly: true,    // ← browser JavaScript CANNOT read this cookie
                      //   prevents XSS attacks from stealing the token
  sameSite: 'lax',  // ← CSRF protection: only sends on same-site requests
  maxAge: 8 * 60 * 60 * 1000  // ← cookie expires in 8 hours (milliseconds)
});

// The token is ALSO returned in the JSON response body
// so the frontend can store it in memory (not localStorage) if preferred
res.json({ token, user });

Why httpOnly?

If a website has an XSS vulnerability (Cross-Site Scripting), malicious JavaScript injected into the page can read localStorage and steal tokens stored there. With httpOnly: true, the cookie is invisible to JavaScript — only the browser itself sends it. This protects the token even if XSS occurs.

Step 5: Sending the Token on Subsequent Requests

After login, the browser automatically sends the cookie on every request to the same domain. The user doesn't need to do anything.

1. Login: Browser → POST /api/auth/login { username, password } Server ← HTTP 200 + Set-Cookie: token=eyJ... httpOnly 2. Any subsequent request: Browser → GET /api/projects Cookie: token=eyJ... ← auto-sent by browser 3. Logout: Browser → POST /api/auth/logout Server ← HTTP 200 + Set-Cookie: token=; Max-Age=0 ← clears the cookie

Step 6: Verifying the Token

File: src/middleware/auth.js

function requireAuth(req, res, next) {
  // Accepts token from cookie (browser) OR Authorization header (API clients)
  const token = req.cookies?.token
    || req.headers.authorization?.split(' ')[1];

  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  try {
    req.user = jwt.verify(token, JWT_SECRET);
    // Re-check the DB to catch disabled accounts mid-session
    const user = checkActive.get(req.user.id);
    if (!user || user.is_active === 0 || user.is_deleted === 1) {
      return res.status(401).json({ error: 'Account disabled' });
    }
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
}

The Change Password Flow

function changePassword(userId, { current_password, new_password }) {
  if (!new_password || new_password.length < 6) {
    throw new ServiceError(400, 'New password must be at least 6 characters');
  }
  const user = userRepo.findById(userId);
  // Must verify the current password before allowing a change
  if (!bcrypt.compareSync(current_password, user.password_hash)) {
    throw new ServiceError(401, 'Current password is wrong');
  }
  const hash = bcrypt.hashSync(new_password, 10);
  userRepo.updatePassword(userId, hash);
  // Also sets must_change_password = 0 in the DB
  return { ok: true };
}
