🧪 Testing

The test suite verifies that the backend works correctly without you having to manually click through everything. There are 5 test files, each focused on one concern. This lesson explains the tools, the structure, and every test.

The Two Testing Tools

Jest

The test runner and assertion library. It finds *.test.js files, runs them, and reports which tests passed or failed. Provides test(), expect(), describe(), beforeAll(), etc.

Supertest

Makes real HTTP requests against your Express app in memory. No network port is opened — it connects directly to the Express instance. Fast, isolated, no server needed.

// How Supertest works — one line to make a request and get the response
const request = require('supertest');
const app = require('../src/app');  // no server.js — just the Express app

const res = await request(app)
  .post('/api/auth/login')
  .send({ username: 'ali', password: 'secret' });

expect(res.status).toBe(200);

Run Command

npm test          # runs: jest --runInBand
# --runInBand means: run test files one at a time (sequential)
# Without it, files could run in parallel and share database state → chaos

tests/helpers.js — The Shared Setup

File: tests/helpers.js

Every test file needs: an isolated database, an Express app connected to it, and a way to create users and tokens. This helper provides all three.

// ── setupTestApp() ─────────────────────────────────────────────
// Creates a completely fresh, isolated environment for one test file.
function setupTestApp() {
  // 1. Clear Node's module cache. Without this, all test files would share
  //    the same db connection (the first require() is cached forever).
  jest.resetModules();

  // 2. Create a temp directory with a unique name (e.g. /tmp/it-pm-test-a3f2/)
  //    so each test file's database never collides with another's.
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'it-pm-test-'));
  process.env.DATA_DIR = tmpDir;

  // 3. Re-require app and db (fresh, pointing to the temp dir)
  const app = require('../src/app');
  const db = require('../src/db/schema');
  const { JWT_SECRET } = require('../src/middleware/auth');
  const jwt = require('jsonwebtoken');

  return { app, db, jwt, JWT_SECRET };
}

// ── seedUser() ─────────────────────────────────────────────────
// Inserts a user directly into the database, bypassing the API.
// WHY: creating users via POST /api/users requires an admin token,
// which creates a circular dependency in auth tests.
// bcrypt cost=1 is used (instead of 10) so tests don't slow down from hashing.
function seedUser(db, { username, password = 'testpass123', role, section_id = null }) {
  const hash = bcrypt.hashSync(password, 1);  // cost=1, fast for tests
  const result = db.prepare('INSERT INTO users (username, password_hash, role, section_id) VALUES (?, ?, ?, ?)')
    .run(username, hash, role, section_id);
  return result.lastInsertRowid;
}

// ── makeToken() ────────────────────────────────────────────────
// Signs a JWT directly, without going through POST /api/auth/login.
// WHY: most tests need an authenticated user but aren't testing the login flow.
// Bypassing login is faster and keeps tests independent of the auth code.
function makeToken(jwt, JWT_SECRET, { id, username, role, section_id = null }) {
  return jwt.sign({ id, username, role, section_id }, JWT_SECRET);
}

The key insight: Each test file calls setupTestApp() in beforeAll(), which resets the module cache and creates a new database in a temp folder. This means the 5 test files never interfere with each other, even if they run sequentially and share the same Node process.

The Five Test Files

auth.test.js

Tests for login, logout, /me, and change-password flows
9 tests
  • Login with correct credentials returns token and user
  • Login with wrong password returns 401
  • Login with unknown username returns 401 (not 404 — no username enumeration)
  • Login with missing password returns 400
  • Login with disabled account returns 403
  • GET /me without token returns 401
  • GET /me with valid token returns user info
  • GET /me with invalid (tampered) token returns 401
// From auth.test.js — the two-step login-then-use-token test:
test('GET /me with valid token returns user info', async () => {
  // Step 1: actually log in to get a real token
  const loginRes = await request(app)
    .post('/api/auth/login')
    .send({ username: 'active_user', password: 'correct123' });
  const token = loginRes.body.token;

  // Step 2: use the token in the Authorization header
  const res = await request(app)
    .get('/api/auth/me')
    .set('Authorization', `Bearer ${token}`);

  expect(res.status).toBe(200);
  expect(res.body).toMatchObject({ username: 'active_user', role: 'super_admin' });
});

  • purchase_admin, it_head, section_head can list purchases; tender_admin and contract_admin cannot

  • contract_admin can read/create contracts; purchase_admin and tender_admin cannot

// From roles.test.js — the authed() helper keeps each test one line
function authed(method, url, role) {
  return request(app)[method](url).set('Authorization', `Bearer ${tokens[role]}`);
}

describe('Projects – create', () => {
  test('super_admin can create a project', async () => {
    const res = await authed('post', '/api/projects', 'super_admin')
      .send({ title: 'Test Project', section_id: 1 });
    expect(res.status).toBe(201);
  });

  test('it_head cannot create a project', async () => {
    const res = await authed('post', '/api/projects', 'it_head')
      .send({ title: 'Denied', section_id: 1 });
    expect(res.status).toBe(403);
  });
});

soft-delete.test.js

Verifies the 10-minute deletion window and soft-delete visibility
4 tests
  • A fresh project (just created) can be deleted — returns 200, row still exists with is_deleted=1
  • A project aged 11 minutes cannot be deleted — returns 409 "older_than_10_min"
  • A soft-deleted project does not appear in GET /api/projects
  • A project aged 9 minutes (boundary) can still be deleted
// From soft-delete.test.js — time travel trick:
// Can't wait 10 real minutes in a test, so we manipulate the DB directly
function ageProject(projectId, minutesAgo) {
  db.prepare(
    `UPDATE projects SET created_at = datetime('now', '-${minutesAgo} minutes') WHERE id = ?`
  ).run(projectId);
}

test('project older than 10 minutes cannot be deleted', async () => {
  const id = await createProject('Old Project');
  ageProject(id, 11);  // simulate 11 minutes ago

  const res = await request(app)
    .delete(`/api/projects/${id}`)
    .set('Authorization', `Bearer ${token}`);

  expect(res.status).toBe(409);
  expect(res.body.error).toBe('older_than_10_min');

  // White-box check: verify the row is still NOT deleted in the DB
  const row = db.prepare('SELECT is_deleted FROM projects WHERE id = ?').get(id);
  expect(row.is_deleted).toBe(0);
});

White-box vs black-box testing: A black-box test only looks at what the API returns. A white-box test also queries the database directly to verify what was written. The soft-delete test uses white-box to confirm that "soft" means the row still exists (not physically deleted).

audit-trail.test.js

Verifies that status changes are correctly recorded in status_history
5 tests
  • Creating a project writes an initial status_history row (from_status=null, to_status='not_started')
  • Changing a project's status adds a new history row with the old and new values
  • Multiple status changes accumulate — history is never overwritten
  • Updating a project WITHOUT changing status does NOT add a history row (no-op skipped)
  • GET /:id/history returns rows newest-first, with changed_by_username joined in
// From audit-trail.test.js — queries DB directly to count history rows
function getHistory(projectId) {
  return db.prepare('SELECT * FROM status_history WHERE entity_type = ? AND entity_id = ? ORDER BY id')
    .all('project', projectId);
}

test('each status change appends a new row — history accumulates', async () => {
  const project = await createProject('Many Changes', 'not_started');

  for (const status of ['in_progress', 'on_hold', 'completed']) {
    await request(app)
      .put(`/api/projects/${project.id}`)
      .set('Authorization', `Bearer ${token}`)
      .send({ status });
  }

  const history = getHistory(project.id);
  expect(history).toHaveLength(4);  // 1 creation + 3 changes
  expect(history[3].to_status).toBe('completed');
});

Testing Concepts You'll Meet Everywhere

async / await in tests

HTTP requests are asynchronous. Without await, the test would check the result before the request finished.

// ❌ Wrong — res is a Promise, not the actual response
test('...', () => {
  const res = request(app).get('/api/projects');
  expect(res.status).toBe(200);  // fails! res.status is undefined
});

// ✅ Correct — await pauses until the HTTP response arrives
test('...', async () => {
  const res = await request(app).get('/api/projects');
  expect(res.status).toBe(200);  // correct!
});

makeToken vs real login

Most tests use makeToken() to get a JWT without going through the login endpoint. This keeps tests focused on what they're actually testing, not on the login flow.

// Only auth.test.js uses the real login — because IT is testing the login flow.
// All other test files use makeToken() directly:
token = makeToken(jwt, JWT_SECRET, { id, username: 'admin', role: 'super_admin' });

// Then attach it on requests:
.set('Authorization', `Bearer ${token}`)

Test Isolation — Why It Matters

If test A creates a record and test B depends on that record existing, then test B fails whenever it runs alone. Good tests are independent. Notice in section-isolation.test.js:

// The delete test creates its OWN project to delete,
// instead of deleting projectInA (which other tests might need)
test('section_head A can delete their own project', async () => {
  // Create a fresh project just for this test
  const created = await request(app)
    .post('/api/projects')
    .set('Authorization', `Bearer ${tokenA}`)
    .send({ title: 'To be deleted by A' });
  // ↑ not reusing projectInA, because deleting it would break visibility tests
  ...
});

How to Add a New Test

Pick the right file for the feature you're testing, then add a test() call:

// Example: testing that archiving a project hides it from default listing
test('archived project does not appear without archived=1 param', async () => {
  // 1. Create a project
  const createRes = await request(app)
    .post('/api/projects')
    .set('Authorization', `Bearer ${token}`)
    .send({ title: 'To Archive', section_id: 1 });
  const id = createRes.body.id;

  // 2. Archive it
  await request(app)
    .patch(`/api/projects/${id}/archive`)
    .set('Authorization', `Bearer ${token}`)
    .send({ archive: true });

  // 3. List without archived=1 — should not appear
  const res = await request(app)
    .get('/api/projects')
    .set('Authorization', `Bearer ${token}`);

  expect(res.body.map(p => p.id)).not.toContain(id);
});
