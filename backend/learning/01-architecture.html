🏗️ Architecture Overview

Before touching a single line of code, you need a mental map of how everything fits together. This lesson explains the folder structure and the four-layer architecture that every request travels through.

The Big Picture

The backend follows Clean Architecture — a pattern where code is split into layers, each with a single responsibility. A request enters at the top (HTTP), passes through each layer, and a response bubbles back up.

┌──────────────────────────────────────────────────────┐ │ HTTP Request arrives │ └──────────────────────┬───────────────────────────────┘ │ ┌────────▼────────┐ │ Route Handler │ ← src/routes/*.js │ (what URL?) │ decides which code to run └────────┬────────┘ │ ┌────────▼────────┐ │ Middleware │ ← src/middleware/auth.js │ (allowed in?) │ checks JWT, roles └────────┬────────┘ │ ┌────────▼────────┐ │ Service │ ← src/services/*.js │ (is it valid?) │ business logic + rules └────────┬────────┘ │ ┌────────▼────────┐ │ Repository │ ← src/repositories/*.js │ (get the data) │ SQL queries only └────────┬────────┘ │ ┌────────▼────────┐ │ SQLite DB │ ← data/it_pm.db │ │ the actual data store └─────────────────┘

The Four Layers — In Plain English

Layer 1 — Routes (src/routes/)

Think of routes as a receptionist at the front desk. They answer: "Which URL did you request? What HTTP method?" and direct you to the right place. They read data from the request (body, URL params, query string) and send the response back. They do NOT contain any business logic.

Routes own the HTTP layer. They speak HTTP: status codes, cookies, headers. Every other layer is completely unaware of HTTP.

Layer 2 — Middleware (src/middleware/)

Middleware is code that runs in between the request arriving and your handler running. Think of it as a security checkpoint. The auth middleware asks: "Do you have a valid JWT token? Are you allowed to do this?". If not, it stops the request right there.

Layer 3 — Services (src/services/)

Services are the brain of the application. They answer: "Is this request valid? Does this user have permission? What rules apply here?". They coordinate between repositories and enforce business rules. This is where you'd find logic like: "only the section_head of section 3 can edit section 3's projects".

Services never touch HTTP. They don't know about req, res, status codes, or cookies. They just receive plain values and return plain values (or throw a ServiceError).

Layer 4 — Repositories (src/repositories/)

Repositories are the database librarians. They only know one thing: SQL. They receive values, run queries, and return raw rows. No business logic. No role checks. Just data access. This separation means if you ever switch from SQLite to PostgreSQL, you only change the repositories.

Why Split Into Layers?

Problem without layersHow layers solve it
SQL mixed with role checks mixed with HTTP responsesEach concern lives in exactly one place
Can't test business logic without starting a serverServices can be tested with plain function calls
Changing the database breaks route handlersRoutes never touch SQL — only repositories do
Duplicated validation logic everywhereOne service = one place for all rules

The "One File Per Domain" Pattern

Notice that routes, services, and repositories all have the same set of files: auth, projects, purchases, tenders, contracts, etc.. This is intentional. Each domain (projects, purchases…) has exactly one file in each layer. When you work on "purchases", you always know which three files to look at:

  • src/routes/purchases.js — the HTTP endpoints
  • src/services/purchaseService.js — the business rules
  • src/repositories/purchaseRepository.js — the SQL

The Database: SQLite

Unlike MySQL or PostgreSQL, SQLite is a single file on your disk (data/it_pm.db). There's no separate database server to install or configure. The database is the file. This makes development and deployment extremely simple.

SQLite is great for this app's scale. If you had thousands of concurrent users writing at the same time, you'd switch to PostgreSQL. The repository layer makes that migration straightforward.
