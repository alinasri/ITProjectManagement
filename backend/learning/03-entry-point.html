🚀 Entry Point: server.js & app.js

When you run npm start, Node.js starts at one file and executes it top-to-bottom. Let's trace exactly what happens from that first line of code to a running server.

How Node.js Starts

Running node src/server.js tells Node: "start executing this file". Node reads it line by line. When it hits a require(), it pauses, loads and runs that other file, and resumes. This is how the whole application bootstraps.

CommonJS modules: This project uses require() to import and module.exports to export. This is the older Node.js module system (as opposed to ES Modules which use import/export).

server.js — The Starting Gun

File: src/server.js

const app = require('./app');  // loads app.js and gets the Express app

// process.env.PORT — if the hosting environment set a PORT variable, use it.
// Otherwise fall back to 3000 for local development.
const PORT = process.env.PORT || 3000;

// app.listen() binds to a TCP port and starts accepting connections.
// The callback fires once, confirming the server is ready.
app.listen(PORT, () => console.log(`Server running on port ${PORT}`));

That's it. Three lines. Its only job is to start the server. All the real configuration is in app.js.

Why Is This Separate From app.js?

Tests need to import the Express app to make test requests against it — but they should not bind to a real port (port conflicts, test isolation). By keeping port-binding in server.js, tests can import app.js directly and use supertest to make fake HTTP requests without a network.

// tests/auth.test.js — imports app, never starts a real server
const request = require('supertest');
const app = require('../src/app');

test('POST /api/auth/login', async () => {
  const res = await request(app).post('/api/auth/login').send({ username: 'admin', password: '...' });
  expect(res.status).toBe(200);
});

app.js — The Express Configuration

File: src/app.js

const express = require('express');      // the web framework
const cookieParser = require('cookie-parser'); // reads Cookie header
const cors = require('cors');              // cross-origin headers
const path = require('path');             // Node built-in: file paths
const fs = require('fs');               // Node built-in: file system

const app = express();  // creates the Express application object

// ── GLOBAL MIDDLEWARE ──────────────────────────────────────────
// These run on EVERY request before any route handler

app.use(cors({ origin: true, credentials: true }));
app.use(express.json());
app.use(cookieParser());

// ── ROUTES ────────────────────────────────────────────────────
// Mount each domain's router at its URL prefix

app.use('/api/auth',         require('./routes/auth'));
app.use('/api/sections',     require('./routes/sections'));
app.use('/api/users',        require('./routes/users'));
app.use('/api/personnel',   require('./routes/personnel'));
app.use('/api/ongoing-tasks',require('./routes/ongoingTasks'));
app.use('/api/projects',    require('./routes/projects'));
app.use('/api/custom-columns',require('./routes/customColumns'));
app.use('/api/purchases',   require('./routes/purchases'));
app.use('/api/tenders',     require('./routes/tenders'));
app.use('/api/contracts',   require('./routes/contracts'));
app.use('/api/report',      require('./routes/report'));

// ── PRODUCTION STATIC FILES ────────────────────────────────────
// Serve the compiled React app if the /public folder exists

const publicDir = path.join(__dirname, '../public');
if (fs.existsSync(publicDir)) {
  app.use(express.static(publicDir));
  app.get('*', (req, res) => res.sendFile(path.join(publicDir, 'index.html')));
}

module.exports = app;  // export so server.js and tests can import it

Understanding app.use()

app.use(prefix, router) tells Express: "for any request whose URL starts with prefix, hand it to this router". The prefix is stripped before passing to the router.

Request: GET /api/projects/deadline-changes app.use('/api/projects', projectsRouter) ↓ strips the prefix projectsRouter sees: GET /deadline-changes

What Does express() Return?

express() creates an application object. It is a function itself (the request handler) plus a bunch of methods for configuration. When Node's HTTP server receives a connection, it calls this function with (req, res). Express then runs your middleware and route handlers.

The Three Global Middleware

1. CORS

CORS (Cross-Origin Resource Sharing) is a browser security feature. By default, a browser running on localhost:5173 (the React dev server) is blocked from fetching from localhost:3000 (your API) because they are different "origins". The CORS middleware adds special headers telling the browser: "this is allowed".

app.use(cors({ origin: true, credentials: true }));
// origin: true  → allow any origin (fine for dev; restrict in production)
// credentials: true → allow cookies to be sent cross-origin

2. express.json()

When a POST or PUT request has a JSON body, the body arrives as raw text bytes. express.json() reads those bytes and parses them into a JavaScript object, putting it in req.body.

// Without express.json(): req.body is undefined
// With express.json(): req.body = { title: "Network upgrade", status: "in_progress" }

3. cookieParser()

HTTP requests include a Cookie header that looks like: token=eyJ...; other=value. cookieParser() parses that string into an object at req.cookies.

// Without cookieParser(): req.cookies is undefined
// With cookieParser(): req.cookies = { token: 'eyJ...' }

The Package.json Scripts

// package.json
{
  "scripts": {
    "start": "node src/server.js",      // production — run once
    "dev":   "nodemon src/server.js",   // development — auto-restarts on file changes
    "test":  "jest --runInBand"         // run all tests sequentially
  }
}
  • npm start — runs the server once. If you change a file, you must restart manually.
  • npm run dev — nodemon watches your files and automatically restarts whenever you save a change.
  • npm test — runs the Jest test suite. --runInBand means "one test at a time" (prevents database conflicts).

What Happens at require('./db/schema')?

When the repositories are loaded (which happens when routes are loaded via require('./routes/projects')), they all import the database. The database module (src/db/schema.js) connects to SQLite and creates all the tables if they don't exist yet. This happens once, at startup.

The database connection is a singleton — no matter how many files do require('../db/schema'), Node's module cache ensures it only runs once and returns the same connection object each time.
