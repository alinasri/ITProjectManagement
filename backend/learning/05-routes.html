🗺️ Routes Layer

Routes define the surface area of your API — which URLs exist, which HTTP methods they accept, who is allowed to access them, and what service function handles each one.

HTTP Verbs — What They Mean

HTTP requests always include a verb (method). Conventions:

VerbMeaningExample
GETFetch data (read-only)GET /api/projects — list projects
POSTCreate a new resourcePOST /api/projects — create project
PUTReplace/update a resource (full update)PUT /api/projects/5 — update project 5
PATCHPartial update (change one field)PATCH /api/projects/5/archive — toggle archive
DELETERemove a resourceDELETE /api/projects/5 — soft-delete project 5

Express Router

Each route file creates an Express Router — a mini-app that handles a subset of URLs. The router is then mounted in app.js at a prefix:

// src/routes/projects.js
const express = require('express');
const router = express.Router();  // mini-app for /api/projects/*

// Define routes on the router, not on app directly
router.get('/', ...);
router.post('/', ...);
router.put('/:id', ...);

module.exports = router;  // exported so app.js can mount it
// src/app.js
app.use('/api/projects', require('./routes/projects'));
// All routes in projects.js are now under /api/projects/

The Full Projects Router

File: src/routes/projects.js

const express = require('express');
const { requireAuth, requireRole } = require('../middleware/auth');
const projectService = require('../services/projectService');
const { wrap } = require('../utils/routeUtils');

const router = express.Router();

// GET /api/projects?section_id=2&archived=0
// → list all projects visible to this user
router.get('/',
  requireAuth,                               // ← must be logged in
  wrap(200, req => projectService.list(req.user, req.query))
);

// POST /api/projects   body: { title, section_id, status, ... }
// → create a new project
router.post('/',
  requireAuth,
  requireRole('super_admin', 'section_head'),  // ← only these roles
  wrap(201, req => projectService.create(req.user, req.body))
);

// ⚠️ IMPORTANT: This must come BEFORE /:id
// Otherwise Express would match "deadline-changes" as the :id parameter
router.get('/deadline-changes',
  requireAuth,
  requireRole('super_admin', 'it_head'),
  wrap(200, () => projectService.getDeadlineChanges())
);

// GET /api/projects/5/history
// → audit trail for project 5
router.get('/:id/history',
  requireAuth,
  wrap(200, req => projectService.getHistory(req.params.id))
);

// PUT /api/projects/5   body: { title, status, ... }
// → update project 5
router.put('/:id',
  requireAuth,
  requireRole('super_admin', 'section_head'),
  wrap(200, req => projectService.update(req.user, req.params.id, req.body))
);

// PATCH /api/projects/5/archive   body: { archive: true }
// → archive or unarchive project 5
router.patch('/:id/archive',
  requireAuth,
  requireRole('super_admin', 'section_head'),
  wrap(200, req => projectService.archive(req.user, req.params.id, req.body.archive))
);

// DELETE /api/projects/5
// → soft-delete project 5 (only within 10 minutes of creation)
router.delete('/:id',
  requireAuth,
  requireRole('super_admin', 'section_head'),
  wrap(200, req => projectService.softDelete(req.user, req.params.id))
);

module.exports = router;

Reading Data From the Request

There are four places where request data can arrive. Here's how to read each one:

SourceExample URL/RequestHow to read it
URL paramsGET /api/projects/5req.params.id → "5"
Query stringGET /api/projects?section_id=2req.query.section_id → "2"
Request bodyPOST with JSON bodyreq.body.title → "Network Upgrade"
CookiesCookie: token=eyJ...req.cookies.token → "eyJ..."

URL params are always strings. req.params.id gives you "5" (string), not 5 (number). SQLite is flexible about this, but be aware when doing strict comparisons.

URL Parameters: :id

The colon in /:id means "match anything here and put it in req.params.id":

router.get('/:id', (req, res) => {
  console.log(req.params.id);
  // GET /api/projects/42 → prints "42"
  // GET /api/projects/abc → prints "abc"
});

The wrap() Utility Explained

Without wrap(), every route handler would need its own try/catch:

// Without wrap() — repetitive boilerplate
router.get('/', requireAuth, (req, res) => {
  try {
    const result = projectService.list(req.user, req.query);
    res.status(200).json(result);
  } catch (err) {
    res.status(err.status || 500).json({ error: err.message });
  }
});

// With wrap() — same thing, one line
router.get('/', requireAuth, wrap(200, req => projectService.list(req.user, req.query)));

The 200 in wrap(200, ...) is the HTTP status code to send on success. 201 means "Created" (used for POST). 200 means "OK" (used for GET/PUT/PATCH/DELETE).

Why Route Order Matters

Express matches routes in the order they are defined. This creates a trap with URL params:

// ❌ WRONG ORDER — "/deadline-changes" would match /:id first
router.get('/:id', ...);            // matches FIRST — "deadline-changes" becomes req.params.id
router.get('/deadline-changes', ...); // never reached

// ✅ CORRECT ORDER — specific paths before param paths
router.get('/deadline-changes', ...); // checked first — matches the literal string
router.get('/:id', ...);            // only reached if not a literal match

Rule: Always define specific paths (like /deadline-changes) before wildcard param paths (like /:id) in the same router.

What Routes Should NOT Do

Routes should be thin. They should not contain business logic. Compare these two approaches:

// ❌ BAD — business logic leaking into the route
router.post('/', requireAuth, (req, res) => {
  if (!req.body.title) return res.status(400).json({ error: 'Title required' });
  if (req.user.role === 'section_head') req.body.section_id = req.user.section_id;
  const result = projectRepo.create(req.body);  // skipping the service!
  res.status(201).json(result);
});

// ✅ GOOD — route is thin, all logic is in the service
router.post('/', requireAuth, requireRole('super_admin', 'section_head'),
  wrap(201, req => projectService.create(req.user, req.body))
);
