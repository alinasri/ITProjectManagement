🗄️ Database & Schema

The database is the single source of truth for everything in the system. This lesson covers every table, every column, how they relate to each other, and how the schema evolves safely over time via migrations.

SQLite — A File Is the Database

Unlike PostgreSQL or MySQL, SQLite needs no server. The entire database lives in one file: backend/data/it_pm.db. The app opens it on startup and reads/writes to it directly. This makes setup trivial and backups as simple as copying one file.

// src/db/schema.js
const { DatabaseSync } = require('node:sqlite');

// DATA_DIR can be overridden by environment variable (for Docker volumes, etc.)
// Default: backend/data/
const DATA_DIR = process.env.DATA_DIR || path.join(__dirname, '../../data');

const db = new DatabaseSync(path.join(DATA_DIR, 'it_pm.db'));

// WAL mode: allows multiple readers while one writer is active (better concurrency)
db.exec("PRAGMA journal_mode = WAL");

// SQLite ignores foreign keys by default — this must be turned on explicitly
db.exec("PRAGMA foreign_keys = ON");

How Migrations Work

The schema in db/schema.js uses CREATE TABLE IF NOT EXISTS for the current state of the database. But what about changes to existing databases? You can't drop and recreate tables — users would lose all their data. Migrations handle this.

// The pattern: check if already done → run if not → stamp as done
function runMigration(id, fn) {
  if (hasRun(id)) return;  // already applied — skip
  fn();                   // run the change
  stamp(id);              // record it as done
}

// Example: migration 009 adds a new column to an existing table
// ALTER TABLE ... ADD COLUMN is safe — existing rows get the default value
runMigration('009_add_due_date_to_projects', () => {
  db.exec(`ALTER TABLE projects ADD COLUMN due_date TEXT`);
});

SQLite limitation: SQLite cannot modify or drop columns via ALTER TABLE. To change a column definition (like adding a new value to a CHECK constraint), you must create a new table with the new definition, copy the data, drop the old table, and rename the new one. See migration 003_add_registry_admin_roles for an example of this pattern.

Initial Seed Data

On a brand-new database, the app seeds five sections and one admin account so the system is immediately usable:

// Only runs when the sections table is empty (fresh database)
const existing = db.prepare('SELECT COUNT(*) as c FROM sections').get();
if (existing.c === 0) {
  // Five default sections (in Persian)
  [
    'تحول دیجیتال و مدیریت داده‌ها',
    'شبکه و امنیت',
    'پشتیبانی',
    'ساماندهی اطلاعات اکتشافی',
    'طراحی و توسعه',
  ].forEach(name => insertSection.run(name));

  // Default admin account: username=admin, password=admin123
  // must_change_password=1 forces a password reset on first login
  db.prepare(`INSERT INTO users (username, password_hash, role, must_change_password) VALUES (?, ?, 'super_admin', 1)`)
    .run('admin', bcrypt.hashSync('admin123', 10));
}

Key Design Decisions Explained

Why TEXT for dates, amounts, and phone numbers?

SQLite stores dates as TEXT (there is no native DATE type). Using TEXT is consistent, and comparison/sorting works naturally for ISO-format dates (2024-03-15). Amounts are also stored as TEXT because currency formatting (commas, currency symbols, decimal places) is handled in the frontend.

Why are purchases/tenders/contracts linked to sections via a join table rather than a direct FK?

Projects belong to exactly one section. But a purchase order might apply to multiple sections (e.g. hardware bought for the Support and Network sections). The join table (purchase_sections) models this many-to-many relationship cleanly.

Why does status_history.entity_id have no FK?

If entity_id were a FK to projects.id (for example), then the ON DELETE behavior would be a problem. With ON DELETE CASCADE, soft-deleting a project would wipe its history — defeating the purpose. With ON DELETE SET NULL, you'd lose which project the history belongs to. No FK is the right choice: the audit trail is never cleaned up.
