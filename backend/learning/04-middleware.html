🛡️ Middleware: Auth & Guards

Middleware is one of Express's most powerful concepts. Once you understand it, everything else in the framework clicks. Let's learn what it is, how it works, and then read the actual auth middleware in this project.

What Is Middleware?

Middleware is a function that has access to req, res, and a special function called next. It runs between the incoming request and your route handler.

// This IS middleware — it's just a function with (req, res, next)
function myMiddleware(req, res, next) {
  console.log('A request came in!');
  next();  // pass control to the next thing in the chain
}

app.use(myMiddleware);  // register it — runs on every request

A middleware function must do one of two things:

  • Call next() — "I'm done, pass this to the next middleware"
  • Send a response — "I'm stopping this request here" (res.json(), res.status().json())

If a middleware neither calls next() nor sends a response, the request hangs forever. The browser will time out waiting.

Middleware Chains

You can stack multiple middleware on one route. Express runs them in order, left to right:

router.get('/', requireAuth, requireRole('super_admin'), actualHandler);
//          ↑ runs 1st     ↑ runs 2nd                     ↑ runs 3rd
Request arrives │ ▼ ┌──────────────┐ │ requireAuth │ → token missing? → sends 401, STOPS here └──────┬───────┘ token ok? → sets req.user, calls next() │ ▼ ┌──────────────────────┐ │ requireRole('admin') │ → wrong role? → sends 403, STOPS here └──────┬───────────────┘ right role? → calls next() │ ▼ ┌─────────────────┐ │ route handler │ → runs the actual logic, sends response └─────────────────┘

The Auth Middleware File

File: src/middleware/auth.js

const jwt = require('jsonwebtoken');
const db = require('../db/schema');

const JWT_SECRET = process.env.JWT_SECRET || 'change-this-secret-in-production';

// Prepared once at module load — fast for every request
const checkActive = db.prepare('SELECT is_active, is_deleted FROM users WHERE id = ?');

// ── requireAuth ────────────────────────────────────────────────
// Guards routes that need a logged-in user.
// Extracts the JWT from cookie or Authorization header,
// verifies it, checks the DB, then attaches user info to req.
function requireAuth(req, res, next) {
  // Look for token in cookie first, then Authorization header
  // Authorization header format: "Bearer eyJ..."
  const token = req.cookies?.token
    || req.headers.authorization?.split(' ')[1];

  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  try {
    // jwt.verify() does two things:
    // 1. Checks the cryptographic signature (was this issued by us?)
    // 2. Checks the expiry date (is it still valid?)
    req.user = jwt.verify(token, JWT_SECRET);
    // req.user is now: { id, username, role, section_id, iat, exp }

    // Extra DB check: what if the account was disabled AFTER the token was issued?
    // The token could still be valid (not expired) but the account is locked.
    const user = checkActive.get(req.user.id);
    if (!user || user.is_active === 0 || user.is_deleted === 1) {
      return res.status(401).json({ error: 'Account disabled' });
    }

    next();  // token valid + account active → proceed
  } catch {
    // jwt.verify() throws if signature is wrong or token is expired
    res.status(401).json({ error: 'Invalid token' });
  }
}

// ── requireRole ────────────────────────────────────────────────
// A "higher-order function" — takes role names and RETURNS middleware.
// Must be used after requireAuth (which sets req.user).
function requireRole(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user?.role)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

module.exports = { requireAuth, requireRole, JWT_SECRET };

Understanding requireRole — A Higher-Order Function

Notice that requireRole is called in the route definition, not just referenced:

// requireAuth is passed as a reference (no parentheses)
router.get('/', requireAuth, ...);

// requireRole('super_admin') is CALLED — it returns a new function
router.get('/', requireAuth, requireRole('super_admin'), ...);

This pattern is called a higher-order function: a function that returns another function. It lets you pass arguments (the allowed roles) to a middleware factory and get back a configured middleware.

// What requireRole('super_admin', 'section_head') actually does:
const roleMiddleware = requireRole('super_admin', 'section_head');
// roleMiddleware is now:  (req, res, next) => { ... checks ['super_admin','section_head'] ... }

// These two lines are equivalent:
router.post('/', requireAuth, requireRole('super_admin', 'section_head'), handler);
router.post('/', requireAuth, roleMiddleware, handler);

Where Does req.user Come From?

When a user logs in, the server creates a JWT (a signed token) containing their user data:

// Created at login (authService.js):
const token = jwt.sign(
  { id: 5, username: 'ali', role: 'section_head', section_id: 2 },
  JWT_SECRET,
  { expiresIn: '8h' }
);

// On the next request, requireAuth decodes it:
// req.user = { id: 5, username: 'ali', role: 'section_head', section_id: 2, iat: ..., exp: ... }

After requireAuth runs, req.user is available in every middleware and route handler that comes after it in the chain.

Optional Chaining (?.) Explained

req.cookies?.token
// Same as: req.cookies ? req.cookies.token : undefined
// Safe — doesn't throw if req.cookies is null or undefined

req.headers.authorization?.split(' ')[1]
// Safe — doesn't throw if authorization header is missing

Security: Why Check the DB Every Request?

JWT tokens are valid until they expire (8 hours). If an admin disables a user's account, the user still has a valid token. Without the DB check, they could keep making requests for up to 8 hours after being disabled.

The extra checkActive query adds a tiny overhead but ensures that account disabling takes effect immediately.

The prepared statement checkActive is compiled once at module load, not on every request. SQLite stores the compiled query plan and just substitutes the user ID each time. This makes it very fast.
