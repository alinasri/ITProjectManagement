🔄 Request Lifecycle

Let's follow one real request from the browser all the way to the database and back. Understanding this flow means understanding the entire backend.

The Scenario

The frontend calls: GET /api/projects?section_id=2

The user is logged in. They want to see all projects for section 2. Here's every single thing that happens.

Step-by-Step Flow

Step 1 — The Browser Sends an HTTP Request

The browser (or frontend app) sends a network packet to localhost:3000. It looks like this:

GET /api/projects?section_id=2 HTTP/1.1
Host: localhost:3000
Cookie: token=eyJhbGciOiJIUzI1NiJ9...  ← the JWT stored in a cookie

HTTP is just text. A request is a text message sent over a network connection. Express reads it and turns it into a JavaScript object called req.

Step 2 — Node.js Receives It

server.js is listening on port 3000. When a connection arrives, Node calls the Express app object, which starts running middleware in order.

Step 3 — Global Middleware Runs (app.js)

Before any route handler runs, Express passes the request through three global middleware functions configured in app.js:

MiddlewareWhat it doesResult on req
cors()Adds CORS headers so the browser allows cross-origin requestsadds response headers
express.json()Reads the request body (if any) and parses JSONpopulates req.body
cookieParser()Reads the Cookie headerpopulates req.cookies

Step 4 — Route Matching

Express checks every mounted route group until it finds a match for /api/projects. In app.js:

app.use('/api/projects', require('./routes/projects'));

Express strips /api/projects and passes the request to the projects router, where it matches GET /:

// src/routes/projects.js
router.get('/', requireAuth, wrap(200, req => projectService.list(req.user, req.query)));

Step 5 — Route-Level Middleware: requireAuth

Before the route handler runs, requireAuth middleware executes. It:

  1. Looks for a JWT in req.cookies.token
  2. Calls jwt.verify(token, JWT_SECRET) — this checks the cryptographic signature and expiry
  3. Queries the database to make sure the account isn't disabled
  4. Sets req.user = { id, username, role, section_id }
  5. Calls next() to proceed

If the token is missing, expired, or the account is disabled, requireAuth sends 401 Unauthorized and the request stops here. The route handler never runs.

Step 6 — The Route Handler Calls the Service

The wrap() utility runs the inner function and catches errors. It calls:

projectService.list(req.user, req.query)
// req.user  → { id: 5, username: 'ali', role: 'section_head', section_id: 2 }
// req.query → { section_id: '2' }

Step 7 — Service Applies Business Rules

Inside projectService.list():

function list(user, { section_id, archived }) {
  const showArchived = archived === '1' ? 1 : 0;

  // Business rule: section_head can ONLY see their own section
  const sectionId = user.role === 'section_head'
    ? user.section_id          // forced to their section
    : section_id;              // admins can pass any section_id

  return projectRepo.enrich(projectRepo.findAll({ sectionId, showArchived }));
}

Step 8 — Repository Runs SQL

The repository receives plain values and executes prepared SQL statements:

// Prepared statement (compiled once at startup)
findBySection: db.prepare('SELECT * FROM projects WHERE section_id = ? AND is_archived = ? AND is_deleted = 0 ORDER BY row_order, id')

// At call time, bind the actual values
findBySection.all(2, 0)  // → returns array of plain JS objects (rows)

Step 9 — Repository Enriches the Data

The raw rows don't include the list of responsible people or custom field values. The enrich() function runs two more SQL queries (one for responsibles, one for custom values) and attaches them to each project. This avoids N+1 queries (explained fully in Lesson 07).

// Returns: Array of project objects, each with extra fields:
[
  {
    id: 1, title: 'Network Upgrade', status: 'in_progress',
    responsibles: [{ id: 3, name: 'Sara' }],    ← added by enrich()
    custom_values: [{ column_id: 1, value: 'High' }]  ← added by enrich()
  },
  ...
]

Step 10 — Data Returns Up the Stack

The enriched array bubbles back up: repository → service → route handler. The wrap() utility catches the return value and sends it as JSON:

res.status(200).json([{ id: 1, title: 'Network Upgrade', ... }, ...])

Step 11 — The Browser Receives the Response

HTTP/1.1 200 OK
Content-Type: application/json

[{"id":1,"title":"Network Upgrade","status":"in_progress","responsibles":[...]}]

The Complete Flow Visualized

Browser │ │ GET /api/projects?section_id=2 │ Cookie: token=eyJ... ▼ Node.js (port 3000) │ ├─ cors() adds CORS headers ├─ express.json() populates req.body ├─ cookieParser() populates req.cookies │ ├─ Route match: GET /api/projects → routes/projects.js │ ├─ requireAuth verifies JWT, sets req.user │ ├─ wrap(200, fn) catches errors, sends response │ ├─ projectService.list() │ applies business rules (section_head restriction) │ ├─ projectRepo.findAll() │ runs: SELECT * FROM projects WHERE section_id = 2 │ ├─ projectRepo.enrich() │ runs: SELECT responsibles for all projects (1 query) │ runs: SELECT custom_values for all projects (1 query) │ └─ res.status(200).json([...]) sends HTTP response with JSON body │ ▼ Browser receives: [{"id":1,"title":"Network Upgrade",...}]

What Happens When Something Goes Wrong?

At any step, if an error is thrown:

// In a service:
if (!project) throw new ServiceError(404, 'Project not found');

// wrap() catches it:
function wrap(statusCode, fn) {
  return (req, res) => {
    try {
      res.status(statusCode).json(fn(req));
    } catch (err) {
      res.status(err.status || 500).json({ error: err.message });
      // sends: HTTP 404 { "error": "Project not found" }
    }
  };
}

Key Terms to Remember

TermWhat it means
reqThe request object — contains everything the browser sent (body, params, cookies, user)
resThe response object — methods to send data back (res.json, res.status, res.cookie)
next()Tells Express to proceed to the next middleware in the chain
middlewareA function that runs between request and handler, either calls next() or ends the request
route handlerThe final function that processes the request and sends a response
