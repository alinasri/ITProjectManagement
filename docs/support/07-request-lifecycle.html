07. A Request, End to End

Every previous page covered one layer in isolation. This page traces a single user action — creating a new project — through all of them, in order, so you can see how the pieces actually connect.

The scenario

A section_head user is on their SectionDashboard page (page 02), fills in a project title, and clicks "Add Project".

Step by step, with real code

1. The component calls the API function

Inside a form submit handler in a section's project list (the general shape, simplified):

const handleAdd = async () => {
  const r = await projects.create({ title, section_id: sectionId });
  setList(prev => [...prev, r.data]);
};

projects.create is defined in frontend/src/api/index.js as (data) => api.post('/projects', data) — it's a thin wrapper, the real work happens in the shared Axios instance from page 03.

2. Axios sends the request, the cookie rides along automatically

Because api was created with baseURL: '/api' and withCredentials: true, this becomes an actual HTTP POST to /api/projects with the JSON body { title, section_id }, and the browser automatically attaches the httpOnly login cookie (page 06).

3. (Dev only) Vite silently forwards it

In development the browser is talking to localhost:5173, but nothing there understands /api routes. Vite's proxy config (page 01) catches the path prefix and relays the entire request to localhost:3000, where Express is actually listening. In production this step doesn't exist — Express is already serving the page the request came from.

4. Express's middleware pipeline runs

First the three global middlewares from app.js run (CORS check, JSON body parsing, cookie parsing — page 04). Then, because the matched route is router.post('/', requireAuth, requireRole('super_admin', 'section_head'), handler) in routes/projects.js, two more middlewares run in order:

  • requireAuth reads the cookie, verifies the JWT, attaches the decoded payload to req.user, and confirms the account is still active.
  • requireRole('super_admin', 'section_head') checks req.user.role is in that list. Since this user is a section_head, it passes.

If either had failed, the route handler below would never run.

5. The route handler talks to the database

backend/src/routes/projects.js

router.post('/', requireAuth, requireRole('super_admin', 'section_head'), (req, res) => {
  const { title, section_id } = req.body;
  const sectionId = req.user.role === 'section_head' ? req.user.section_id : section_id;

  const result = db.prepare(
    `INSERT INTO projects (title, section_id, status, ...) VALUES (?, ?, ?, ...)`
  ).run(title.trim(), sectionId, 'not_started', /* ... */);

  const projectId = result.lastInsertRowid;
  recordStatusChange('project', projectId, null, 'not_started', req.user.id);

  const project = db.prepare('SELECT * FROM projects WHERE id = ?').get(projectId);
  res.status(201).json(enrichProjects([project])[0]);
});

Notice it doesn't blindly trust section_id from the request body — for a section_head, it forces the project into their own section (req.user.section_id, taken from the verified JWT, not anything the client could fake). This is a server-side authorization check, distinct from the role check the middleware already did.

After inserting the project, it immediately writes an audit-trail row via recordStatusChange (page 05), then re-reads the full row back out (to pick up any database defaults) and sends it back as JSON with status 201 Created.

6. The response flows back and updates the screen

Axios resolves the promise with the parsed JSON in r.data. Back in the component, setList(prev => [...prev, r.data]) adds the new project to React state — which makes React re-render the list, and the new project appears, with no page reload at any point in this entire round trip.

Key idea: One button click crosses roughly seven boundaries — component, Axios, network/proxy, Express middleware (x2), route handler, database, and back. Nothing here is magic; it's the exact same handful of patterns from pages 02-06, just chained together in sequence.

Self-check

Why does the route handler override section_id with req.user.section_id for a section_head, instead of trusting whatever the frontend sent?

Because the request body is fully controllable by whoever is making the HTTP call — a malicious or buggy client could send any section_id it wants. The server can't trust client-supplied data for authorization decisions; it must rely on the verified identity in req.user, which came from a cryptographically signed JWT it issued itself.

At which exact step would the request stop if the user's JWT cookie had expired?

Inside requireAuth, step 4 — jwt.verify would throw because the token's expiresIn window has passed, the catch block sends 401 Invalid token, and requireRole and the route handler never run. The frontend's Axios interceptor would then see the 401 and redirect to /login.

Where in this flow does the audit trail (status_history) get written, relative to the actual INSERT?

Immediately after, in the same route handler, using the lastInsertRowid returned by the INSERT. Both happen as separate SQL statements within one request — there's no database transaction wrapping them, so in theory if the server crashed between the two statements you could get a project with no matching history row, but this app doesn't guard against that edge case.

Try it: pick a different action in the app (e.g. creating a purchase, or changing a contract's status) and find its matching route file under backend/src/routes/. Does it follow the same pattern — middleware, then INSERT/UPDATE, then optionally a status_history write, then a fresh SELECT before responding?

It should — this request/response shape (validate → mutate → audit → re-fetch → respond) repeats across purchases.js, tenders.js, and contracts.js almost verbatim, which is exactly why understanding one route file deeply lets you read all the others quickly.
