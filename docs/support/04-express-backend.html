04. Express Backend

What Node.js and Express actually are, what middleware means, and how a route file turns an HTTP request into a database query.

What is Node.js?

Node.js is a runtime that lets JavaScript run outside a browser — on a server, with access to things browsers deliberately don't allow, like the filesystem and raw network sockets. Without Node, JavaScript could only run inside a webpage.

What is Express?

Node by itself can open a network port and read raw HTTP requests, but doing that for every route by hand is tedious. Express is a small library that adds a clean layer on top: define a path and an HTTP method, and Express runs your function when a matching request arrives — and gives you parsed request bodies, route parameters, and a simple way to send back JSON.

In this app — the server entry point

backend/src/app.js

const app = express();

app.use(cors({ origin: true, credentials: true }));
app.use(express.json());
app.use(cookieParser());

app.use('/api/auth', require('./routes/auth'));
app.use('/api/sections', require('./routes/sections'));
app.use('/api/projects', require('./routes/projects'));
// ...one line per resource

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Server running on port ${PORT}`));

This file does three things: creates the Express app, registers middleware that runs on every request, then mounts one router per resource under a URL prefix.

What is middleware?

Middleware is a function that sits in the request's path before it reaches your actual route handler. Each middleware gets the request, the response, and a next() function — calling next() passes control to whatever comes next; not calling it (and instead sending a response) stops the request right there. This is exactly how access control works: a middleware can inspect the request and short-circuit it with a 401 before any business logic runs.

Three middlewares run on every request, registered with app.use(...) at the top of app.js:

  • cors(...) — allows the browser (running on a different port in dev) to make requests to this server at all.
  • express.json() — parses an incoming JSON request body into req.body, so route handlers don't have to.
  • cookieParser() — parses the Cookie header into req.cookies, which is how the login token is read.

Auth middleware — gatekeeping specific routes

backend/src/middleware/auth.js

function requireAuth(req, res, next) {
  const token = req.cookies?.token || req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  try {
    req.user = jwt.verify(token, JWT_SECRET);
    // ...check the user is still active in the DB
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
}

function requireRole(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user?.role)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

Unlike the three global middlewares above, these two are only attached to specific routes that need them — see the next section. requireRole(...roles) is a function that returns a middleware, which is why it's called like requireRole('super_admin', 'section_head') right inside a route definition: it builds a custom checker for that exact route's allowed roles.

What is a route handler, and what is REST?

REST is just a convention for designing APIs: a URL identifies a resource (e.g. /api/projects means "the collection of projects", /api/projects/5 means "project number 5"), and the HTTP method describes the action — GET to read, POST to create, PUT/PATCH to update, DELETE to remove.

In this app — a full route file

backend/src/routes/projects.js

router.get('/', requireAuth, (req, res) => {
  const projects = db.prepare(
    'SELECT * FROM projects WHERE is_archived = 0 AND is_deleted = 0 ORDER BY id'
  ).all();
  res.json(enrichProjects(projects));
});

router.post('/', requireAuth, requireRole('super_admin', 'section_head'), (req, res) => {
  const { title, section_id } = req.body;
  if (!title?.trim()) return res.status(400).json({ error: 'Title required' });
  const result = db.prepare(
    'INSERT INTO projects (title, section_id, status) VALUES (?, ?, ?)'
  ).run(title.trim(), section_id, 'not_started');
  res.status(201).json({ id: result.lastInsertRowid });
});

(Simplified from the real file for clarity — the actual version also records an audit-trail entry and handles assigning responsible personnel; see page 05 and page 07 for the full version.)

Notice the middleware chain: router.get('/', requireAuth, (req, res) => {...}) — Express runs requireAuth first, and only calls the final handler function if requireAuth called next(). The POST route adds a second middleware, requireRole(...), so creating a project requires both being logged in and having the right role.

Key idea: A request flows through a pipeline: global middleware (CORS, JSON parsing, cookies) → route-specific middleware (requireAuth, requireRole) → the route handler itself, which talks to the database and sends a JSON response. Any step can stop the pipeline early by responding instead of calling next().

Self-check

What would happen if a route handler forgot to call res.json(...) or res.send(...) at all?

The HTTP request would simply hang — Express never closes the response, so the browser (or Axios) waits indefinitely until it eventually times out. Every code path through a route handler must end by sending some response.

In router.post('/', requireAuth, requireRole('super_admin', 'section_head'), handler), what happens if a logged-in purchase_admin user calls this route?

requireAuth passes (they have a valid token), so it calls next(). Then requireRole('super_admin', 'section_head') checks req.user.role — 'purchase_admin' isn't in that list, so it responds with 403 Forbidden and never calls next(), meaning the actual route handler never runs.

Why is express.json() registered with app.use(...) instead of being added individually to each route?

app.use(...) (with no path) registers middleware that runs on every single incoming request, regardless of which route it eventually matches. Since nearly every route needs req.body parsed from JSON, it makes sense to do it once globally rather than repeat it per route.

Try it: open backend/src/routes/auth.js and find a route that does not use requireAuth. Why doesn't it need it?

POST /login has no requireAuth — that's the whole point, since a user calling it doesn't have a token yet. It's the route that produces the token in the first place.
