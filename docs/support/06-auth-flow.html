06. Authentication Flow

What a JWT is, why it's stored in an httpOnly cookie, and the full path from typing a password to seeing a protected dashboard.

What is JWT authentication?

HTTP is stateless — by default, the server has no memory of who you are between requests. JWT (JSON Web Token) solves this by packaging identity information (user id, role, etc.) into a signed string. The server signs it with a secret key when you log in; on every later request, the server re-checks that signature to prove the token wasn't tampered with, without needing to store any session data itself.

A JWT has three dot-separated parts (header, payload, signature). Anyone can read the payload (it's just base64, not encrypted) — but only someone with the secret key can produce a signature that verifies, so a token can't be forged or modified.

What is an httpOnly cookie, and why use one?

The token has to live somewhere in the browser so it can be sent on every request. Two common choices: in JavaScript-accessible storage (like localStorage), or in a cookie marked httpOnly. This app uses the cookie approach specifically because an httpOnly cookie cannot be read by JavaScript at all — only the browser itself attaches it to requests. If malicious code ever got injected into the page (an XSS attack), it still couldn't steal the token, because it has no API access to it.

Step 1 — the server checks the password

backend/src/routes/auth.js

const user = db.prepare('SELECT * FROM users WHERE username = ?').get(username);
if (!user || !bcrypt.compareSync(password, user.password_hash)) {
  return res.status(401).json({ error: 'Invalid username or password' });
}

Passwords are never stored in plain text — password_hash is a one-way bcrypt hash. bcrypt.compareSync hashes the submitted password the same way and compares hashes; it's intentionally slow, which makes brute-forcing it impractical.

Step 2 — the server signs a token and sets a cookie

const token = jwt.sign(
  { id: user.id, username: user.username, role: user.role, section_id: user.section_id },
  JWT_SECRET,
  { expiresIn: '8h' }
);

res.cookie('token', token, { httpOnly: true, sameSite: 'lax', maxAge: 8 * 60 * 60 * 1000 });

The token carries everything a later request will need to know about the user — their id and role mainly — so route handlers don't have to look the user up by id on every single request just to know their role (though requireAuth still re-checks is_active/is_deleted from the database, in case an admin disabled the account after the token was issued).

Step 3 — every later request proves who you are automatically

Because the cookie was set by the server and the frontend's Axios instance uses withCredentials: true (see page 03), the browser attaches the cookie to every subsequent request to the API without any extra code. The backend's requireAuth middleware (see page 04) reads it back out:

const token = req.cookies?.token || req.headers.authorization?.split(' ')[1];
if (!token) return res.status(401).json({ error: 'Unauthorized' });
req.user = jwt.verify(token, JWT_SECRET);

jwt.verify both checks the signature is valid and that the token hasn't expired. If either fails, it throws, and the catch block in requireAuth responds 401.

Step 4 — the frontend re-checks auth on every page load

Refreshing the browser wipes all React state, but not the cookie. That's why AuthContext calls auth.me() on mount (see page 03) — it's asking the server "given my cookie, who am I?" so the app can restore user without forcing a fresh login every time the page reloads.

Logout and 401 handling

POST /api/auth/logout calls res.clearCookie('token'), deleting the cookie. Separately, the Axios response interceptor (page 03) watches for any 401 response and force-redirects to /login — this is what happens automatically if a token expires mid-session.

Key idea: The frontend never manages the token directly — it can't even read it. The browser silently attaches the httpOnly cookie to every request; the backend is the only thing that ever signs, reads, or invalidates it.

Self-check

If you open your browser's DevTools console on this app and type document.cookie, will you see the JWT token?

No — that's precisely the point of marking the cookie httpOnly. It's sent automatically by the browser on requests, but is completely invisible to any JavaScript running on the page, including DevTools console scripts.

What stops someone from editing the JWT payload (e.g. changing their own role to super_admin) and using the modified token?

The signature. Any change to the payload changes what the signature should be, and the server recomputes/verifies the signature using JWT_SECRET (which only the server knows) on every request via jwt.verify. A tampered token fails verification and is rejected with 401.

An admin disables a user's account mid-session by setting is_active = 0. Does that user's still-valid, unexpired JWT immediately stop working?

Yes — requireAuth doesn't just trust the JWT payload; after verifying it, it runs SELECT is_active, is_deleted FROM users WHERE id = ? and rejects the request if the account is now disabled, even though the token itself is still cryptographically valid.

Try it: open frontend/src/context/AuthContext.jsx and find where auth.me() is called. What happens to loading while that request is in flight, and why does App.jsx care?

loading starts true and is set false in the .finally() after auth.me() resolves or rejects. App.jsx's ProtectedLayout and RequireAuth components check loading first and render a spinner instead of redirecting — without this, the app would briefly flash the login page for already-logged-in users before the auth check finishes.
