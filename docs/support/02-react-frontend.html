02. React Frontend

What a component is, how this app is broken into pieces, and how the browser decides which screen to show.

What is a component?

A React component is just a JavaScript function that returns a description of some UI (written in a syntax called JSX, which looks like HTML mixed into JavaScript). React calls that function, gets back the description, and turns it into real DOM elements on the page. When the component's data changes, React calls the function again and updates only what changed.

function StatCard({ label, value }) {
  return (
    <div className="rounded-lg p-4 bg-gray-900">
      <p>{label}</p>
      <p>{value}</p>
    </div>
  );
}

label and value here are props — values passed into a component from whoever uses it, the same way arguments are passed into a function.

In this app — pages vs. components

This codebase draws a clear line between two kinds of components, both under frontend/src/:

A page like PurchasesPage.jsx is built by combining smaller components — it might render a PageHeader, a FilterBar, a list of rows each with a StatusBadge, and a ConfirmDialog for deletes. None of those pieces know about purchases specifically — they're generic and reused everywhere.

What is routing, and what is React Router?

A normal website has multiple actual HTML pages — clicking a link loads a new file from the server. A single-page app (SPA) like this one loads one HTML file, and a library called React Router watches the URL in the address bar and decides which component to show, entirely inside the browser, without a full page reload.

How a route gets "locked"

Look at this real snippet from App.jsx:

function RequireAuth({ roles }) {
  const { user, loading } = useAuth();
  if (loading) return null;
  if (!user) return <Navigate to="/login" replace />;
  if (roles && !roles.includes(user.role)) return <Navigate to="/" replace />;
  return <Outlet />;
}

RequireAuth is a "gatekeeper" component. It's placed as a wrapper around protected routes; before rendering the real page, it checks whether someone is logged in (user) and whether their role is in the allowed list. If not, it redirects instead of rendering. <Outlet /> is React Router's way of saying "render whichever nested route matched."

Key idea: Routing is just "look at the URL, pick a component." Access control is layered on top by wrapping routes in components like RequireAuth that decide to redirect instead of render.

Self-check

What's the practical difference between something living in pages/ vs. components/?

A page is tied to a specific URL/route and typically fetches its own data from the API. A component has no route of its own — it's a reusable visual building block that pages assemble together (and it usually receives data via props rather than fetching it itself).

If a section_head user types /admin into the address bar, what happens?

RequireAuth wraps that route with roles={['super_admin']}. Since section_head isn't in that list, the check !roles.includes(user.role) is true, and the user is redirected to / instead of seeing the admin dashboard.

What does :sectionId in the path /section/:sectionId/* mean?

It's a dynamic URL segment. Visiting /section/3/projects matches this route and makes 3 available inside SectionDashboard via React Router's useParams() hook, so the component knows which section's data to load.

Try it: open frontend/src/pages/PurchasesPage.jsx and find one component from components/ that it imports and renders.

Look at the top import list and the JSX returned — you should find things like PageHeader, StatusBadge, ConfirmDialog, or FilterBar being used to assemble the page.
