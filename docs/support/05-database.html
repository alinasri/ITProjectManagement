05. Database & Schema

What a relational database actually is, every table in this app, how they connect, and the audit-trail pattern used throughout.

What is a relational database, a table, and a foreign key?

A relational database stores data in tables — think spreadsheets, where each row is one record and each column is one field. A foreign key is a column in one table that holds the id of a row in another table, creating a link between them (e.g. a project's section_id points at a row in the sections table). ON DELETE CASCADE means "if the row being pointed to is deleted, automatically delete this row too" — used here so deleting a section cleans up its projects.

A join table (also called a many-to-many table) solves a specific problem: when one project can have several responsible people, and one person can be responsible for several projects, you can't fit that into a single foreign-key column. Instead, a separate table (project_responsibles) stores one row per (project, person) pairing.

In this app — no ORM, just SQL

Many Node apps use an ORM (a library that lets you write JavaScript instead of SQL, e.g. Sequelize or Prisma). This app deliberately skips that — it uses Node's built-in node:sqlite module directly, writing raw SQL with placeholders (?) that get safely substituted with real values, which prevents SQL-injection attacks.

backend/src/db/schema.js

const { DatabaseSync } = require('node:sqlite');
const db = new DatabaseSync(path.join(DATA_DIR, 'it_pm.db'));

db.exec("PRAGMA journal_mode = WAL");
db.exec("PRAGMA foreign_keys = ON");

The soft-delete and audit-trail pattern

Instead of permanently deleting rows with SQL DELETE, every main entity table (projects, ongoing_tasks, purchases, tenders, contracts, users) has an is_deleted flag column. "Deleting" just sets is_deleted = 1, and every list query filters with WHERE is_deleted = 0. This gives a recovery window — the actual route enforces it's only allowed within 10 minutes of creation (see backend/src/routes/projects.js, the DELETE /:id handler). is_archived works similarly for hiding completed work from the main view without deleting it.

The status_history table is a generic audit log shared by five different entity types. Every time a project's status or due date changes, a row is inserted recording the old value, the new value, who changed it, and when — via two small helper functions:

backend/src/db/helpers.js (used from routes)

recordStatusChange('project', projectId, oldStatus, newStatus, req.user.id);
recordFieldChange('project', projectId, 'due_date', oldDueDate, newDueDate, req.user.id);

Because entity_type + entity_id identify which row in which table changed (rather than a foreign key to one specific table), this one table can log history for projects, tasks, purchases, tenders, and contracts without needing five separate history tables.

Key idea: The schema favors simplicity over normalization tricks — flag columns instead of a separate "deleted items" table, one shared history table instead of five, and join tables only where a true many-to-many relationship exists.

Self-check

Why does a project need a separate project_responsibles table instead of just a responsible_id column on projects?

Because more than one person can be responsible for a single project (and one person can be responsible for many projects) — that's a many-to-many relationship, which a single foreign-key column can't represent. (In fact, an earlier version of this schema did have a single responsible_id column — there's a migration in schema.js that moves existing data out of it into the new join table.)

What does WHERE is_deleted = 0 in a query actually accomplish, given the row still physically exists in the database?

It hides "deleted" rows from normal queries (lists, lookups) without destroying the data, which makes a temporary undo possible. The data is only truly gone if something later runs an actual SQL DELETE, which this app doesn't do for these entities.

Why can status_history store history for five different kinds of entities in one table instead of needing five tables?

Because it identifies the row generically with entity_type (a string like 'project') plus entity_id (that row's id), instead of a strict foreign key to one specific table. The tradeoff is the database itself can't enforce that entity_id actually exists in the right table — that's left to the application code.

Try it: open backend/src/db/schema.js and find the block of code (around line 153) that migrates responsible_id into the join tables. What SQL keyword does it use to avoid duplicate rows if it runs twice?

INSERT OR IGNORE — if a (project_id, personnel_id) pair already exists in the join table, the insert is silently skipped instead of throwing a unique-constraint error, which makes this migration safe to run every time the server starts.
