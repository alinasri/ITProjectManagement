08. Running & Deploying

What actually happens when you type npm run dev, and how the same codebase becomes a single Docker image in production.

Development: two terminals

There is no single command that starts both halves of the app — you run two separate Node processes, each with its own package.json and its own npm run dev script.

Backend

backend/package.json

cd backend
npm install
npm run dev

npm run dev here runs nodemon, a tool that watches every .js file under backend/ and automatically restarts the Node process whenever one changes — so you don't have to manually stop and rerun the server after every edit. It starts Express listening on http://localhost:3000 (or whatever PORT is set to).

Frontend

frontend/package.json

cd frontend
npm install
npm run dev

This runs Vite's dev server on http://localhost:5173. Vite compiles your React/JSX on the fly and pushes updates into the browser almost instantly (hot module replacement) without a full page reload when you edit a component.

The proxy that connects them

frontend/vite.config.js

export default defineConfig({
  plugins: [react(), tailwindcss()],
  server: {
    proxy: {
      '/api': 'http://localhost:3000',
    },
  },
});

Without this, a fetch('/api/projects') made from a page served on port 5173 would try to hit port 5173 itself — there's nothing listening there for /api. The proxy rule tells Vite: "any request path starting with /api, forward it to port 3000 instead, and relay the response back." This is purely a development convenience; it doesn't exist in the production build at all (covered below).

Production: one Docker image, one process

Docker lets you package an application plus everything it needs to run (Node itself, dependencies, files) into one portable image. This project uses a multi-stage build — multiple FROM blocks in one Dockerfile, where later stages can copy specific files out of earlier ones, discarding everything else (like the entire node_modules used only to build the frontend).

Dockerfile

# Stage 1: Build React frontend
FROM node:20-alpine AS builder
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

# Stage 2: Run Express backend (serves API + built frontend)
FROM node:20-alpine
WORKDIR /app
COPY backend/package*.json ./
RUN npm ci --omit=dev
COPY backend/ .
COPY --from=builder /app/frontend/dist ./public

VOLUME /app/data
ENV DATA_DIR=/app/data
ENV NODE_ENV=production
ENV PORT=3000

EXPOSE 3000
CMD ["node", "src/app.js"]

What each stage does

  • Stage 1 ("builder") installs the frontend's dependencies and runs npm run build, which is vite build under the hood — it compiles all the JSX/React code down to plain, optimized HTML/CSS/JS files in frontend/dist/. This stage is thrown away at the end except for that one output folder.
  • Stage 2 starts fresh, installs only the backend's production dependencies (--omit=dev skips nodemon, ESLint, etc. — things only needed while developing), copies the backend source code in, then copies frontend/dist from stage 1 into backend/public.

That last copy is exactly what makes the line you saw in page 01 work: backend/src/app.js checks if a public/ folder exists and, if so, serves it as static files plus falls back to index.html for unmatched routes (so React Router still works for direct URL visits like /admin).

The data volume

VOLUME /app/data plus ENV DATA_DIR=/app/data means the SQLite file lives outside the container's own filesystem, in a Docker-managed volume. This matters because containers are disposable — if you rebuild or replace the container, anything written only inside it is lost. Mounting /app/data as a volume means the database survives container restarts and upgrades.

Running it

docker build -t it-pm .
docker run -p 3000:3000 -v it-pm-data:/app/data it-pm

-p 3000:3000 maps the container's port 3000 to your machine's port 3000. -v it-pm-data:/app/data creates (or reuses) a named volume called it-pm-data and mounts it at /app/data inside the container — this is the actual flag that makes the database persistent.

Environment variables that matter

VariableUsed byPurpose
PORTbackend/src/app.jsWhich port Express listens on (defaults to 3000)
DATA_DIRbackend/src/db/schema.jsWhere the SQLite file is created/read (defaults to backend/data)
JWT_SECRETbackend/src/middleware/auth.js, routes/auth.jsThe key used to sign/verify login tokens — must be set to something real in production, or anyone could forge a valid token using the default fallback value

Key idea: Development optimizes for fast feedback (two servers, instant reload, a proxy gluing them together). Production optimizes for a single deployable, self-contained unit — pre-built static files served by one process, with persistent state pushed out to a volume.

Self-check

If you forget to run the frontend's npm run dev and only start the backend, what would happen if you visited http://localhost:3000 directly in development?

You'd likely get nothing useful — in development there's no backend/public folder (that only gets created by the Docker build step), so the static-file-serving block in app.js wouldn't activate, and visiting the root path wouldn't match any API route either. In dev you're meant to visit the Vite server on port 5173, not Express directly.

Why does the Dockerfile run npm ci instead of npm install?

npm ci installs exactly the versions locked in package-lock.json, deletes any existing node_modules first, and fails if the lockfile and package.json are out of sync — making builds reproducible. npm install can update the lockfile and is meant for active development, not repeatable builds.

What would happen to your data if you ran docker run -p 3000:3000 it-pm (forgetting the -v volume flag), used the app for a while, then removed the container?

Without a mounted volume, /app/data lives only inside that specific container's writable layer. Removing the container deletes that layer — and the entire SQLite database, including all projects, users, and history — along with it.

Try it: open backend/src/middleware/auth.js and find the fallback value for JWT_SECRET if the environment variable isn't set. Why is shipping that default to production risky?

It falls back to the literal string 'change-this-secret-in-production'. Since this is visible in the public source code, anyone could use that exact string to forge a valid JWT for any user id and role they want, if a real deployment forgot to override it with a proper secret via the JWT_SECRET environment variable.
