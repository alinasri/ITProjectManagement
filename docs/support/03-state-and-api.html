03. State & API Calls

How a component remembers things between renders, how that data is shared across the whole app, and how it gets fetched from the server in the first place.

What is "state", and what is a hook?

A component function re-runs every time something changes, but plain JavaScript variables inside a function don't survive between calls. State is React's way of giving a component memory: a value that persists across re-renders, and that — when updated — triggers React to re-render the component with the new value.

Hooks are special functions (always starting with use) that let a plain function component tap into React features like state. The two used constantly in this app:

  • useState(initialValue) — returns [currentValue, setterFunction]. Calling the setter schedules a re-render with the new value.
  • useEffect(fn, deps) — runs fn after the component renders, and re-runs it whenever any value in the deps array changes. It's the standard place to fetch data.

In this app — the fetch-on-mount pattern

Pages across the app (e.g. PurchasesPage.jsx) follow the same shape:

const [list, setList] = useState([]);
const [loading, setLoading] = useState(true);

const fetch = useCallback(async () => {
  try {
    const r = await purchasesApi.list(params);
    setList(r.data);
  } finally {
    setLoading(false);
  }
}, [params]);

useEffect(() => { fetch(); }, [fetch]);

Reading this top to bottom: list starts empty; the moment the component first appears on screen, useEffect runs fetch(); fetch calls the API, and when the response arrives, setList updates state, which makes React re-render the component — this time with real data to display.

What is Context, and why not Redux?

Passing data from a parent component to a deeply nested child by threading it through every component in between ("prop drilling") gets painful fast. Context is React's built-in answer: a value declared once near the top of the tree that any descendant component can read directly, no matter how deep, via a hook.

This app only has two pieces of truly global state — who's logged in, and the list of sections — so it uses plain Context instead of pulling in a state-management library like Redux, which would be more machinery than two values need.

In this app — AuthContext

frontend/src/context/AuthContext.jsx

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    auth.me()
      .then(r => setUser(r.data))
      .catch(() => setUser(null))
      .finally(() => setLoading(false));
  }, []);

  const login = async (username, password) => {
    const r = await auth.login({ username, password });
    setUser(r.data.user);
    return r.data.user;
  };
  // ...
  return (
    <AuthContext.Provider value={{ user, setUser, login, logout, loading }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);

AuthProvider wraps the entire app (see App.jsx) and, on first load, asks the server "who am I?" via auth.me() — relying on the login cookie, covered in page 06. Any component anywhere can then call const { user, login, logout } = useAuth(); to read or change auth state, without it being passed down as props.

SectionsContext.jsx follows the identical pattern for the list of organizational sections, which many pages need.

What is Axios, and how does the frontend call the backend?

Axios is a small library for making HTTP requests from JavaScript — similar to the browser's built-in fetch, but with a few conveniences (automatic JSON parsing, request/response interceptors, a configurable base URL).

In this app — the API client

frontend/src/api/client.js

const api = axios.create({
  baseURL: '/api',
  withCredentials: true,
});

api.interceptors.response.use(
  r => r,
  err => {
    if (err.response?.status === 401 && !onPublicPage) {
      window.location.href = '/login';
    }
    return Promise.reject(err);
  }
);

Every request made through this api instance automatically gets the prefix /api (so a call to projects becomes a request to /api/projects), and withCredentials: true tells the browser to include the login cookie on every request. The interceptor is a hook that runs on every response — here, if any request comes back 401 Unauthorized, it bounces the whole app to the login page.

On top of this raw client, frontend/src/api/index.js defines small named functions per resource, e.g.:

export const purchases = {
  list:   (params) => api.get('/purchases', { params }),
  create: (data)   => api.post('/purchases', data),
  update: (id, data) => api.put(`/purchases/${id}`, data),
};

This is why pages can write the readable purchasesApi.list(params) instead of repeating the URL and method everywhere.

Key idea: Data flows in one direction per cycle: a hook triggers a fetch → Axios sends the HTTP request → the response updates state via a setter → React re-renders with the new state. Context just makes a slice of that state visible app-wide instead of local to one component.

Self-check

Why does calling setList(...) cause the screen to update, but just reassigning a normal variable wouldn't?

React specifically tracks state created via useState. Calling the setter function doesn't just change a value in memory — it tells React "this component needs to re-render," and React re-runs the component function and reconciles the new output with the DOM. A plain variable reassignment has no such hook into React's render cycle.

What problem does Context solve that plain props don't?

It avoids "prop drilling" — manually passing a value through every intermediate component that doesn't even use it, just so a deeply nested component can access it. Context lets any descendant subscribe directly via a hook like useAuth(), regardless of nesting depth.

What does withCredentials: true actually change about a request?

By default, browsers don't send cookies on cross-origin (or even some same-origin-via-proxy) requests unless told to. This flag tells Axios to include the browser's cookies — specifically the JWT login cookie — with every API request, which is what lets the backend recognize who's making the request.

Try it: open frontend/src/context/SectionsContext.jsx and compare it line-by-line to AuthContext.jsx above. What's the one structural difference?

SectionsContext doesn't track a loading flag the same way and exposes a refresh() function instead of login/logout actions — but the shape (useState + useEffect-on-mount + Context.Provider + a custom use... hook) is identical. Once you recognize this shape, you'll recognize it everywhere in the app.
