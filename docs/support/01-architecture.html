01. Big-Picture Architecture

Before looking at any code, it helps to see the shape of the whole system: who talks to whom, and where each piece lives.

What is a "client-server" app?

This app is split into two programs that run separately and talk over HTTP:

  • The client — the React app that runs inside your browser. It draws the screen and reacts to clicks.
  • The server — the Express app that runs on Node.js. It owns the database and decides what data the client is allowed to see or change.

The client never touches the database directly. It always asks the server, over HTTP, and the server is the only thing with a direct line to the SQLite file. This separation is what lets you (in theory) swap the React app for a phone app later without touching the backend at all.

Dev mode: two servers, one proxy

While you're developing, two separate processes run on your machine at once:

graph LR B[Browser] -->|http://localhost:5173| V[Vite Dev Server] V -->|requests starting with /api are proxied| E[Express :3000] E -->|reads/writes| D[(SQLite file
backend/data/it_pm.db)]

Vite serves the React app on port 5173 with instant hot-reload. Express serves the API on port 3000. Because the browser can only easily talk to one origin without extra CORS headers, Vite is configured to forward any request starting with /api straight to Express — see frontend/vite.config.js, covered in page 08.

Production: one server

In production (the Docker build), there's no Vite dev server at all. The React code is compiled ahead of time into static HTML/CSS/JS files, and Express simply serves those files itself alongside the API:

graph LR B[Browser] -->|http://your-server:3000| E[Express :3000] E -->|static files| F[Compiled React build] E -->|/api/*| R[API routes] R --> D[(SQLite file)]

This is why backend/src/app.js has a block that serves a public/ folder if it exists — that folder is the compiled React app, copied there by the Docker build (see backend/src/app.js lines 24-30, and page 08 for the full Docker walkthrough).

Key idea: "frontend" and "backend" aren't just folders — they're two separate running programs. In dev they run on different ports and Vite bridges them; in production they collapse into a single Express process.

Self-check

Why can't the React app query the SQLite database directly?

The database file lives on the server's disk, not in the browser. The browser has no filesystem access to it — it can only reach the database indirectly, by sending HTTP requests to Express, which runs on the same machine as the database file and is the only thing with a direct connection to it.

In development, what does Vite's proxy actually do?

It watches for any browser request whose path starts with /api and silently forwards it to http://localhost:3000 (the Express server), then sends the response back — so the React code can just call /api/projects without needing to know or care that Express is running on a different port.

What's different about how the React code is served in production vs. development?

In development, Vite serves React source files live and recompiles on every change. In production, the React code is pre-compiled once into static files (HTML/CSS/JS) by vite build, and Express just hands those files out like any other static website — no Vite process runs at all.

Try it: open backend/src/app.js and find the line that decides whether to serve the React build. What condition does it check?

It checks fs.existsSync(publicDir) — if a backend/public folder exists (which only happens after the Docker build copies the compiled frontend there), Express serves it as static files and falls back to index.html for any unmatched route (so React Router can handle client-side routes).
